## Verdict

**Fail**

Build-breaking TypeScript errors in the Services admin surface (and dead/prototype code that is still compiled), plus multiple contract/UX inconsistencies that make the page misleading for operators and confusing for non-superusers.

## Findings

### Critical

- **`web` does not typecheck due to Services admin code drift.**
  - `web/src/pages/settings/SettingsAdmin.tsx` passes `onSetHeaderAction` to `ServicesPanel`, but `ServicesPanelProps` does not accept it.
  - `web/src/pages/settings/ServicesPanelStatic.tsx` passes props (`notice`, `onDismissNotice`) that `ServiceDetailRailView` does not accept.
  - Evidence: `web/.node_modules/.bin/tsc -b` reports:
    - `ServicesPanelStatic.tsx(...) ... Property 'notice' does not exist ...`
    - `SettingsAdmin.tsx(...) ... Property 'onSetHeaderAction' does not exist ...`

- **Superuser UX is not gated, but mutation APIs are superuser-gated.**
  - The Settings nav always shows Operations pages (`Services`, `Platform`, `Integration Catalog`, `Grid Test`) via `web/src/pages/settings/settings-nav.ts`.
  - `useSuperuserProbe` exists (`web/src/hooks/useSuperuserProbe.ts`) but is unused; `SettingsAdmin` does not downgrade to browse/read-only mode for non-superusers.
  - Backend mutation endpoints are gated by an email allowlist (`services/pipeline-worker/app/shared/superuser.py`, `supabase/functions/_shared/superuser.ts`), so non-superusers will see admin controls that fail at runtime (403/503).

### Major

- **Admin Services list/read path is inconsistent with the documented “admin-services” endpoint direction.**
  - `ServicesPanel` loads services/functions via direct Supabase `SELECT` (RLS allows `authenticated` to read all rows) rather than calling the superuser-gated `admin-services` endpoint.
  - This diverges from `web-docs/plans/services-scalability-plan.mdx` Task 1 (adds view modes on `admin-services`) and the existence of `GET /admin/services` in both edge + FastAPI.

- **Service base URL is not surfaced correctly in the UI.**
  - `ServiceDetailRailView` displays `{{BLOCKDATA_API_BASE_URL}}` instead of `service.base_url`, and copies placeholder endpoints rather than a real resolved URL.
  - This breaks the operator mental model, especially for multi-environment setups.

- **The “Edit JSON” surface is misleading and unsafe.**
  - The Monaco editor shows `parameter_schema` and `result_schema`, but `saveFunction()` does not submit those fields (only name/type/label/description/entrypoint/http_method/enabled/tags).
  - `handleSaveFunctionJson()` merges arbitrary JSON onto a typed object with a cast, so invalid shapes can cause runtime errors (e.g., `tags` not being an array breaks `.join` inside `functionToDraft`).

- **Platform config values do not drive the Services API base URL.**
  - `PlatformConfigPanel` edits `platform.pipeline_worker_url` (stored via `admin-config`), but `services-panel.api.ts` uses `import.meta.env.VITE_PIPELINE_WORKER_URL` (build-time env) and does not read runtime policy.
  - Outcome: Operators can “save” a URL that does not affect this UI’s behavior.

### Minor

- **Redundant/unused code paths increase drift risk.**
  - `ServicesPanelStatic.tsx` appears to be a prototype/mock but is still compiled and currently breaks typechecking.
  - `ServiceDetailPanel.tsx` exists but is not used by `ServicesPanel` (current UI uses `ServiceDetailRailView`).

- **Realtime refresh strategy is coarse.**
  - `subscribeToServiceChanges()` triggers a full reload of all types/services/functions on any change, including browse mode. This will not scale to the “250+ services” target without debouncing and slice refresh.

## Required Changes Before Implementation (Go/No-Go Gates)

1. **Restore `web` typecheck:** fix or remove `ServicesPanelStatic.tsx` and fix `SettingsAdmin` → `ServicesPanel` prop contract.
2. **Define and enforce the intended access model:**
   - Either gate Operations pages behind `useSuperuserProbe` (hide nav + route guard), or explicitly render them read-only for non-superusers (and suppress mutation controls).
3. **Align read path with backend contracts:**
   - If the `admin-services` endpoint is the source of truth for admin mode, use it (and move read-only browse mode to a separate read endpoint/view).
4. **Fix the function edit contract:**
   - Either restrict editable JSON to the fields actually saved, or extend `saveFunction()` to support `parameter_schema` + `result_schema` (and validate JSON shape before submit).
5. **Remove the base URL placeholder:** display/copy resolved URLs based on `service.base_url + fn.entrypoint` (with safe normalization).
6. **Clarify platform config behavior:** either wire runtime policy into the client (if intended), or label those settings as server-side only and remove misleading defaults.

## Verification Expectations / Acceptance Criteria

- `web` passes `tsc -b` with no errors.
- Non-superusers cannot see or cannot activate admin mutation controls; attempts do not produce confusing partial UI states.
- Services UI shows real `base_url` and copies correct endpoint URLs.
- Editing a function:
  - Either only allows editing fields that are persisted, or persists schema fields with correct validation.
  - Does not crash on malformed input.
- Realtime updates do not trigger full refetch storms at moderate scale; at minimum, debounced refresh is in place.

## Final Recommendation

**No-go** until the Critical items are fixed and the access model is clarified. After that, re-evaluate alignment with `Services Admin Scalability` plan and choose whether admin reads should go through `admin-services` view endpoints.
