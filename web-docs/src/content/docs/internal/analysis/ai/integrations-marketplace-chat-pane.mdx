- Scope reviewed: Kestra integration architecture, backend contract fit, execution readiness

## Verdict

**Fail**

The plan is directionally correct for UX, but not execution-ready for Kestra benchmarking and backend integration. It currently has unresolved schema/contract risks and lacks backend optimization guardrails.

## Findings

### Critical

1. **Catalog table contract mismatch and schema drift risk**
The plan assumes `integration_catalog_items` as the primary runtime source. Existing admin runtime code uses `kestra_plugin_items` and `integration_catalog_items_temp`. This must be reconciled before implementation; otherwise API and UI queries will break per environment.

2. **Public provider detail can expose internal providers**
Index response filters internal providers, but provider detail lookup logic can still resolve internal providers by slug unless explicitly blocked. This is a data exposure risk for internal plugin groups.

3. **No backend benchmarking acceptance criteria**
The plan says this is a benchmark-driven integration but defines no SLOs or measurable acceptance criteria (latency, payload size, query count, cold start budget, concurrency behavior).

### Major

1. **Inefficient aggregation pattern for provider counts**
The plan fetches all enabled rows and counts in application code. This is acceptable at small size but not optimized for growth. Aggregation should be pushed to SQL with `GROUP BY plugin_group`.

2. **No pagination or bounded payload strategy**
Provider detail returns full item sets without pagination. This can become expensive for large providers and harms page-load predictability.

3. **No compatibility mapping to your backend action model**
Your backend has action-oriented integration modeling (`integration_actions`, service/function mapping). The plan does not connect marketplace entries to runtime compatibility, reducing operational value for your actual execution system.

4. **Operational readiness is under-specified**
No logging schema, no endpoint telemetry, no error taxonomy, no rollback checklist for edge-function deploy changes.

### Minor

1. **Mixed workstreams increase blast radius**
Integrations marketplace and assistant layout refactor are independent but bundled in one rollout plan, complicating risk isolation and rollback.

2. **Manual verification only**
Plan includes manual browser checks but no automated contract tests for the new public edge function.

## Gaps, Contradictions, and Ambiguities

- Contradiction: current runtime references `kestra_plugin_items`, while migration history defines `integration_catalog_items` as canonical.
- Ambiguity: no declared canonical source-of-truth table/view for marketplace reads.
- Gap: no statement on whether marketplace should include only externally actionable providers versus all non-internal providers.
- Gap: no explicit auth model decision for read endpoint (authenticated-user scoped vs fully public token-gated).

## Required Changes Before Implementation

1. Decide and document a **single canonical catalog relation** (table/view) for marketplace reads and update all callers consistently.
2. Enforce provider visibility in both index and detail endpoints (`is_internal = false` policy at query layer).
3. Replace in-memory counting with SQL aggregation and add pagination (`limit`, `offset`, deterministic sort) for provider items.
4. Add backend-fit fields in API response for execution readiness (at minimum: action, mapped service/function availability, enabled status).
5. Define benchmark acceptance thresholds (p95 latency, max payload size, expected row counts, acceptable cold start).
6. Add API-level tests for index/detail contract, internal-provider blocking, and auth failure paths.
7. Split rollout into separate releases (A: integrations API+pages, B: assistant pane refactor) with independent rollback steps.

## Verification Expectations

1. Contract tests for:
   - `GET integrations-catalog` returns only allowed providers.
   - `GET integrations-catalog?provider=...` blocks internal providers.
   - Item counts match SQL aggregation baseline.
2. Performance tests:
   - p95 response under target at expected catalog cardinality.
   - Response size under defined limit for index and provider endpoints.
3. Security checks:
   - JWT enforcement behavior matches intended deployment config.
   - No leakage of internal providers/items through alternate query paths.
4. Integration checks:
   - Marketplace item can be mapped to backend execution capability when applicable.
