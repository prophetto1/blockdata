# Load Activation Implementation Assessment

- Source plan: `docs/plans/2026-03-15-load-activation.md`
- Date reviewed: `2026-03-15`
- Reviewer: `Codex`
- Verdict: `Fail`

## Executive Summary

The implementation is real and substantial. The registry migrations landed, the FastAPI connection and load-runs routes exist, the GCS and Arango plugins are wired, the Load UI is present, and the targeted Python test suite passes.

This is no longer a “plan only” situation. It is a functioning vertical slice. What keeps it from a pass is a smaller set of post-implementation correctness gaps:

- successful runs likely leave JSONL artifacts behind in public storage because the implemented artifact path no longer matches the cleanup contract
- partial Arango write failures are recorded as successful items, which can incorrectly stamp a run `complete`
- the Load UI can submit mismatched connections because it does not constrain source/destination connection choices by provider
- the run page can miss initial item state because it subscribes after item creation and does not fetch current run details after submit

## Findings By Severity

### Critical

#### 1. Successful runs likely leave public JSONL artifacts behind because the implementation drifted from the cleanup path contract

The plan’s artifact convention is explicit:

- `docs/plans/2026-03-15-load-activation.md:47`
- `docs/plans/2026-03-15-load-activation.md:2080`

It says artifacts should live at:

- `pipeline/load-artifacts/{run_id}/{item_key_sanitized}.jsonl`

and be cleaned up on successful completion.

The implementation writes a different path:

- `services/platform-api/app/plugins/gcs.py:101`

The download plugin writes:

- `load-artifacts/{context.execution_id}/{object_name_sanitized}.jsonl`

and `step_load()` sets:

- `services/platform-api/app/api/routes/load_runs.py:257`

to:

- `execution_id = f"{run_id}/{item_id}"`

That means the actual object path becomes:

- `load-artifacts/{run_id}/{item_id}/{file}.jsonl`

But cleanup still only lists and removes by:

- `services/platform-api/app/api/routes/load_runs.py:86-90`

and storage uploads return public URLs:

- `services/platform-api/app/infra/storage.py:14-26`

This is a direct contract mismatch. At minimum it breaks fidelity to the promised artifact path. More importantly, it likely prevents successful-run cleanup from removing the actual JSONL files, leaving load artifacts in public storage after completion.

Required change:

- Make artifact writes match the documented contract again by using `run_id` as the artifact directory, not `run_id/item_id`, or
- update cleanup to recurse correctly through nested paths and prove it with an automated test.

### Major

#### 2. Partial Arango write failures are collapsed into successful items and can incorrectly finalize the run as `complete`

The Arango plugin reports partial batch failure as:

- `services/platform-api/app/plugins/arangodb.py:72-76`

It returns `state = "WARNING"` when some documents fail.

But the load route only treats `FAILED` as terminal failure:

- `services/platform-api/app/api/routes/load_runs.py:285-295`

So a partial insert:

- does not raise
- marks the item `complete`
- records `rows_failed`

Then `_maybe_finalize_run()` determines the final run status solely from item rows whose status is `failed`:

- `services/platform-api/app/api/routes/load_runs.py:55-78`

That means a run with partial Arango failures can still be finalized as `complete`.

Required change:

- Treat `WARNING` from the destination plugin as a failed item for run-status purposes, or
- add item-level partial semantics and update finalization logic accordingly.

#### 3. The Load UI can submit invalid source/destination connection combinations

The plan’s wizard implies source-specific and destination-specific connection selection, but the UI currently shows the full connected connection list in both selectors:

- `web/src/pages/LoadPage.tsx:203-207`
- `web/src/pages/LoadPage.tsx:231-235`

There is no provider filtering in `useLoadRun()`:

- `web/src/hooks/useLoadRun.ts:70-74`

and the backend validates only function stage, not connection compatibility:

- `services/platform-api/app/api/routes/load_runs.py:122-131`

So the UI can submit combinations like:

- Arango connection as source
- GCS connection as destination

Those fail later inside plugin execution, not at selection time.

Required change:

- Filter connection options by the selected function’s service/provider, and
- ideally add backend validation that the supplied connection type matches the selected function family.

#### 4. The run UI can miss initial item state because it subscribes after item creation and never fetches the just-created run

After submit, `useLoadRun()` sets `activeRun` and clears `runItems`:

- `web/src/hooks/useLoadRun.ts:143-152`

The realtime subscription starts only after `activeRun` is set:

- `web/src/hooks/useLoadRun.ts:84-118`

and it listens for future `INSERT` and `UPDATE` events on `service_run_items`, but there is no immediate fetch of the existing rows after submit. The hook has `loadRunDetails()`:

- `web/src/hooks/useLoadRun.ts:187-197`

but the page never calls it after creating the run:

- `web/src/pages/LoadPage.tsx:49-63`

Because the backend creates all `service_run_items` before returning from submit, those insert events may be missed by the time the client subscribes. The result is a run page that can start with an empty item list until later updates arrive.

Required change:

- Fetch `GET /load-runs/{run_id}` immediately after submit, or
- have `submit_load()` return the initial item rows.

### Minor

#### 5. The Connections panel omitted the planned “test before save” flow

Task 12 called for an add-connection flow that follows the provider-form pattern: input credentials, test, then save.

The implemented panel allows:

- save on add: `web/src/pages/settings/ConnectionsPanel.tsx:98-117`
- test only after the connection already exists: `web/src/pages/settings/ConnectionsPanel.tsx:143-166`

That is usable, but it is not a tight implementation of the planned UX.

Required change:

- Add a test step in the add form, or
- update the plan/docs to reflect “save then test” as the intended flow.

#### 6. Some canonical naming cleanup remains incomplete outside the runtime slice

The runtime code is now using canonical `service_*` names in the main places that mattered, including:

- `services/platform-api/app/api/routes/admin_services.py`

But a repo scan still finds old `registry_*` naming in some tests and comments, for example:

- `supabase/functions/admin-integration-catalog/index.test.ts`
- `web/src/pages/marketplace/ServiceDetailPage.test.tsx`
- `services/pipeline-worker/app/registry.py`

This is not blocking the Load slice itself, but it leaves residual naming drift in the surrounding codebase.

## Required Changes Before Calling This Complete

1. Fix the artifact path and cleanup mismatch, then prove successful-run cleanup with an automated test.
2. Correct partial-write handling so `WARNING` from the Arango plugin cannot finalize a run as `complete`.
3. Constrain Load connection choices by provider and add backend validation for mismatched connections.
4. Fetch initial run/item state after submit so the UI shows the real item list immediately.
5. Decide whether the Connections page should support pre-save testing or update the plan/docs to reflect the shipped behavior.

## Go / No-Go Recommendation

`No-go` for calling the implementation finished as-is.

The slice is close. The core architecture is present and the Python/backend verification is strong. But the remaining issues are not cosmetic:

- one threatens artifact cleanup and data exposure
- one can misreport partial writes as fully successful runs
- one makes the main Load UI too easy to drive into invalid combinations
- one weakens run progress visibility right after submit

Once those are fixed, this should move quickly into `Conditional Pass` or better.
