# KT High Fidelity Upgrade Plan Assessment

- Source plan: `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md`
- Date reviewed: `2026-03-16`
- Reviewer: `Codex`
- Verdict: `Fail`

## Executive Summary

The direction is right. Raising fidelity in the shared `kt/blockdata` runtime is the correct next step.

The implementation plan is not ready to execute as written. It over-claims what counts as “high fidelity,” omits at least one in-scope low-fidelity core file, contradicts itself about whether `runner.py` is in scope, and contains at least one proposed implementation that is technically wrong in Python.

My answer to “do you agree with the implementation?” is:

- I agree with the goal.
- I do not agree with this implementation plan as currently written.

## Findings By Severity

### Critical

#### 1. The plan’s scope claim is false: it does not cover every in-scope low-fidelity core file

The plan goal says it will upgrade “every `kt/blockdata` core file” to high fidelity, but the task list does not include `property.py`, which the existing assessment explicitly classifies as low fidelity and identifies as a major reduction in scope.

References:

- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:5`
- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:15-1952`
- `web-docs/src/content/docs/assessments/2026-03-16-kt-pre-runtime-build-files-fidelity-assessment.mdx:51-72`

Why this matters:

- The plan’s headline promise is broader than the actual task list.
- A plan that says “every core file” but excludes the lowest-fidelity core property abstraction is not execution-ready.

Required change:

- Either add `kt/blockdata/core/models/property.py` as an explicit task, or narrow the goal statement so it only covers the listed files.

#### 2. The plan contradicts itself about `runner.py`

The architecture section says the worker execution path, including `runner.py`, “is already HIGH fidelity and is not touched.” Later tasks modify `runner.py` repeatedly.

References:

- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:7`
- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:1082-1144`
- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:1672-1749`
- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:1855-1932`

Why this matters:

- This is not a wording nit. It means the plan has not actually stabilized scope.
- Reviewers cannot tell whether `runner.py` is baseline, out of scope, or a planned rewrite target.

Required change:

- Make the scope internally consistent. Either remove `runner.py` changes from the plan or update the architecture section to say the worker path is in scope for targeted fidelity upgrades.

#### 3. The `output.py` implementation is technically wrong and would reject valid dataclass subclasses

The plan proposes enforcing dataclass subclassing via `Output.__init_subclass__`, checking `dataclasses.is_dataclass(cls)` inside subclass creation. That check runs before the `@dataclass` decorator transforms the class, so a valid dataclass subclass would still fail at class definition time.

References:

- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:1754-1850`
- especially `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:1816-1823`

Why this matters:

- This is a concrete implementation bug in a prescribed code block.
- The plan is asking engineers to implement code that will not behave as its own tests assume.

Required change:

- Replace this enforcement strategy with one that works after class decoration, or drop enforcement entirely and keep `Output` as a permissive interface-style base.

### Major

#### 4. Several proposed “high fidelity” APIs are not faithful to Kestra and instead codify local stubs

Multiple tasks claim to upgrade fidelity while introducing interfaces that are still materially different from Kestra’s real surface:

- `task.py` keeps `retry` as `dict[str, Any]`, `worker_group` as `str | None`, and `log_level` as `str | None` rather than real structural analogs.
- `run_context.py` adds `storage_interface()` instead of the Kestra-shaped `storage()` abstraction, and `namespace_kv()` returns a plain dict stub.
- `queue.py` adds pause/resume and a lightweight consumer-group model that is not actually aligned with Kestra’s queue contracts.

References:

- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:205-356`
- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:712-882`
- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:1149-1307`

Why this matters:

- This plan often says “high fidelity” while prescribing “helpful local approximation.”
- Those are not the same thing, and the distinction matters because the stated goal is precise.

Required change:

- For each task, distinguish between:
  1. true Kestra-shape parity,
  2. intentionally compressed compatibility shims,
  3. BlockData-specific host adaptations.

#### 5. Verification assumes a clean green test path, but the plan ignores known prerequisite blockers

The plan repeatedly expects `python -m pytest tests/ -v` to pass after each task. The current repo assessment already records unresolved blockers in package import boundaries and dependency declaration.

References:

- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:191-194`
- repeated full-suite expectations throughout the plan
- `web-docs/src/content/docs/assessments/2026-03-16-kt-new-pulls-assessment.mdx:35-58`
- `web-docs/src/content/docs/assessments/2026-03-16-kt-new-pulls-assessment.mdx:167-184`

Why this matters:

- A good implementation plan must sequence prerequisite environment and packaging fixes before relying on a clean test matrix.
- As written, the plan assumes a verification baseline that the current assessment says does not yet exist.

Required change:

- Add prerequisite setup and verification tasks ahead of the fidelity upgrades, or scope verification to the isolated tests that can actually run cleanly.

#### 6. The plan contains accuracy errors in the very first task

Task 1 says “all 16 values,” but the proposed enum and test set actually include 17 values.

References:

- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:17`
- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:34-39`
- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:126-143`

Why this matters:

- This is an avoidable plan-quality error at the top of the document.
- It reduces confidence that later file-by-file parity claims were counted carefully.

Required change:

- Correct the count and verify every “before/after” claim numerically.

### Minor

#### 7. Some task targets are underspecified relative to the real Kestra source

Examples:

- `run_context_initializer.py` is described as matching three methods plus a rehydration stub, but the actual Kestra class carries more dependency wiring and contextual initialization behavior.
- `storage.py` claims high fidelity from a thin local file implementation, but does not discuss tenant and namespace semantics explicitly.

References:

- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:1312-1452`
- `docs/plans/2026-03-16-kt-high-fidelity-upgrade.md:361-559`

Suggested change:

- Add a short “remaining known divergence” note under each task so the plan does not imply full parity where only partial parity is being built.

## Specific Gaps, Contradictions, and Ambiguity

### Gaps

- No task for `property.py` despite the plan’s “every core file” promise
- No prerequisite step for import-boundary and dependency issues already recorded in the latest runtime assessment
- No explicit separation between true fidelity work and intentional stubs

### Contradictions

- `runner.py` is declared out of scope, then modified in multiple tasks
- The plan claims “high fidelity” while prescribing APIs such as `storage_interface()` and dict-based `namespace_kv()` that are not faithful Kestra analogs

### Ambiguity

- It is unclear whether the plan wants:
  1. exact interface-shape parity with Kestra,
  2. behavioral parity with Pythonic naming,
  3. or staged approximation toward future parity.

Those are different goals and should not be mixed in one task list.

## Verification Expectations and Acceptance Criteria Check

Current assessment against the quality gate:

| Criterion | Status | Notes |
|---|---|---|
| Alignment with product direction and constraints | Partial | The direction is right, but the implementation approach is not consistently faithful. |
| Clear scope boundaries | Fail | The plan says “every core file,” omits `property.py`, and contradicts itself on `runner.py`. |
| API/data contract clarity | Partial | Many code blocks are explicit, but some prescribed APIs are not actually faithful to Kestra. |
| Dependency and sequencing correctness | Fail | The plan ignores prerequisite packaging and import-boundary blockers. |
| Risk handling and rollback strategy | Fail | No rollback or staging strategy for these cross-cutting interface changes. |
| Security/auth implications | Partial | Not central here, but host/runtime boundary changes are not discussed. |
| Operational readiness | Fail | The plan assumes clean repo verification conditions that are not yet in place. |
| Test/verification clarity | Partial | Tests are specified well, but some proposed tests validate the wrong contract or a technically invalid implementation. |

Minimum acceptance criteria for a revised plan:

1. Fix the scope statement so it matches the actual task list.
2. Add or explicitly defer `property.py`.
3. Reconcile whether `runner.py` is in scope.
4. Remove technically invalid implementation prescriptions such as the `Output.__init_subclass__` dataclass check.
5. Add prerequisite tasks for package import decoupling and dependency completeness if full-suite verification is expected.
6. Mark each task as either exact fidelity, staged approximation, or BlockData-specific adaptation.

## Required Changes Before Implementation

1. Rewrite the goal so it no longer promises more than the plan covers.
2. Add the missing `property.py` work or explicitly exclude it with rationale.
3. Normalize the `runner.py` scope language and task list.
4. Replace the broken `output.py` enforcement design.
5. Rework tasks like `task.py`, `run_context.py`, and `queue.py` so their targets reflect actual Kestra parity instead of local placeholder APIs.
6. Add a prerequisite verification section that acknowledges the current repo state recorded in the runtime assessment.

## Go / No-Go Recommendation

`No-go` for implementation from this plan as written.

This is close to a usable plan in structure. It has concrete tasks, failing tests first, and a clear desire to ratchet fidelity file by file. That is good.

But the current version is not yet trustworthy enough to execute. It still confuses true parity with local approximation, misses at least one important in-scope file, contradicts itself on scope, and prescribes at least one implementation that is technically unsound.

The right next step is not to start coding from it. The right next step is to revise the plan so its scope, fidelity claims, and prescribed implementations are defensible.
