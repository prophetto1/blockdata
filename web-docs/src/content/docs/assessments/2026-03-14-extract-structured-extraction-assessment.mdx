# Structured Extraction Plan Assessment

- Source plan: `docs/plans/2026-03-14-extract-structured-extraction.md`
- Date reviewed: `2026-03-14`
- Reviewer: `Codex`
- Verdict: `Fail`

## Summary

The plan is pointed in the right product direction. It keeps extraction post-parse, reuses the platform Vertex path, and narrows scope to document-level and page-level extraction. Those are good decisions.

It is not execution-ready yet. Several task steps conflict with the repo's current storage contract, representation naming, Docling data shape, and schema identity model. If implemented as written, core paths in `run-extract`, schema CRUD, and page-level extraction would either fail outright or produce incorrect results.

## Findings

### Critical

#### 1. The `run-extract` plan uses the wrong artifact contract, so the edge function would not be able to load parsed content

The plan queries `conversion_representations` using `rep_type` and `storage_path`, looks for `docling_document`, and downloads from a `conversion-artifacts` bucket (`docs/plans/2026-03-14-extract-structured-extraction.md:473-504`).

The repo uses different contracts:

- `representation_type`, not `rep_type`
- `artifact_locator`, not `storage_path`
- `doclingdocument_json`, not `docling_document`
- the `documents` bucket, not `conversion-artifacts`

See:

- `supabase/functions/_shared/representation.ts`
- `supabase/functions/conversion-complete/index.ts:119,220-221,370-371`
- `web/src/pages/parseArtifacts.ts:296-301`

This is a hard blocker. Task 4 cannot work as written.

#### 2. The schema identity model collapses distinct schemas and creates cross-user collision risk

The migration makes `schema_uid` the primary key, and the frontend hook derives it from `computeSchemaUid(schemaBody)` before calling `upsert` (`docs/plans/2026-03-14-extract-structured-extraction.md:53,824-839`).

That means schema identity is content-hash-only. In practice, this creates three problems:

- two users cannot safely save the same schema body under different names
- one user cannot keep two schemas with the same body but different labels or different intended use
- the same body saved once as `document` and once as `page` collides, even though `extraction_target` is semantically part of the saved schema

The existing helper confirms the hash is body-only:

- `web/src/lib/extractionSchemaHelpers.ts:176`

This is a data-model blocker for the core schema CRUD flow.

#### 3. The proposed page-text helper does not match the current Docling structure and would drop meaningful content

Task 3 assumes text and table items both expose a plain `text` field. The current Docling table shape does not. Tables are reconstructed from `data.table_cells`, and the existing helper uses a dedicated `tableToText()` path (`web/src/lib/doclingNativeItems.ts:125,198`).

The current document walker also resolves items by `self_ref`, traverses `body` and `furniture`, and handles groups, tables, key-value items, and form items (`web/src/lib/doclingNativeItems.ts:75-182`).

The planned helper does not account for that full structure. As a result, page-level extraction would silently omit table content and could miss other page content routed through non-text nodes.

### Major

#### 4. The frontend integration steps target the wrong project-resolution surface for `/app/extract`

Task 8 wires `useProjectDocuments(focusedProjectId)` and passes `focusedProjectId` into the extraction hooks (`docs/plans/2026-03-14-extract-structured-extraction.md:1019-1037`).

The current project focus hook exposes both `focusedProjectId` and `resolvedProjectId`, and the existing extract workbench already uses `resolvedProjectId` because it is route-aware and falls back safely (`web/src/hooks/useProjectFocus.ts:148-165`, `web/src/pages/useExtractWorkbench.tsx:592-603`).

Switching this workbench back to `focusedProjectId` would regress current routing behavior and can produce empty or stale project state.

#### 5. The plan does not include the automated test work needed for a change of this size

The verification checklist is almost entirely manual. It does not call for tests around:

- the new Docling page assembly helper
- the `run-extract` edge function success and failure paths
- schema UID behavior and CRUD hooks
- the extracted shared schema editor component
- the extract workbench job/result states

This repo already carries tests around the workbench and document table surfaces. A plan that changes migrations, an edge function, shared hooks, and two major pages needs explicit automated coverage before implementation starts.

#### 6. Page-range behavior is still underdefined at the execution boundary

The plan adds `page_range`, but the edge function checks `totalPages > MAX_PAGES` before applying the requested range (`docs/plans/2026-03-14-extract-structured-extraction.md:509-518`).

That means a 200-page document would be rejected even if the user requested pages 1-2. The plan also does not define validation for:

- `start < 1`
- `end < start`
- `end > totalPages`
- a valid range that yields zero assembled page payloads

Those rules need to be written before implementation to avoid inconsistent UX and avoidable support cases.

### Minor

#### 7. The docs output path in the workflow instructions does not match the current workspace

The workflow instructions point to `F:\blockdata-ct\src\content\docs`, but this workspace's docs site lives under `web-docs/src/content/docs`. The plan itself is present in `docs/plans`, not in the docs-site content tree.

This is not a product blocker, but it should be normalized so plan reviews and implementation notes land in the right published docs location.

#### 8. The migration is missing a few execution-readiness details

The plan does not define:

- an FK from `extraction_jobs.source_uid` to `source_documents.source_uid`
- indexes for the main query paths on `project_id`, `source_uid`, and `job_id`
- whether the realtime publication statements should be guarded for repeatable migration behavior

None of these are hard blockers by themselves, but they should be tightened before the migration is treated as production-ready.

## Specific Gaps, Contradictions, and Ambiguity

- The storage and representation contract in Task 4 does not match the current parse pipeline.
- The page helper design does not match the current Docling table and node model.
- The schema persistence design treats content equality as identity, which conflicts with user-level saved-schema behavior.
- The frontend wiring uses a less reliable project identifier than the current workbench already uses.
- The verification plan is not strong enough for a cross-cutting migration plus edge function plus UI rollout.
- The page-range UX is only partially specified.

## Required Changes Before Implementation

1. Rewrite Task 4 to use the current artifact contract:
   `representation_type`, `artifact_locator`, `doclingdocument_json`, and the `documents` bucket.

2. Redesign schema identity:
   use a stable schema row ID or make uniqueness owner-scoped and project-scoped; do not use a body-only hash as the primary key for saved schemas.

3. Replace the proposed page helper design with one based on the live Docling structure:
   preserve table text extraction, furniture traversal, and the current `self_ref` resolution model.

4. Update Task 8 to keep using `resolvedProjectId`, not `focusedProjectId`, unless the routing model is intentionally being changed.

5. Add an automated test plan:
   migration sanity, helper unit tests, edge-function success/failure tests, hook tests, and workbench interaction tests.

6. Define exact page-range validation and oversized-document behavior:
   including whether limits apply to total document pages or only the requested range.

7. Tighten the migration:
   add indexes, add the missing document FK if appropriate, and make publication behavior explicit.

## Verification Expectations

Before this plan should be marked implementation-ready, the revised version should verify all of the following:

- document-level extraction loads artifacts using the live `conversion_representations` contract
- page-level extraction includes table text and does not silently drop Docling content types
- two different users can save the same schema body without collision
- one user can save two schemas with the same body but different names or targets, if the product intends that behavior
- `/app/extract` resolves the active project correctly from the current route and fallback focus state
- edge-function tests cover success, missing artifact, invalid range, oversized page request, and LLM failure paths
- UI tests cover schema save, schema selection, extraction run state, and results rendering

## Final Recommendation

`No-go until revised.`

The plan has good product shape, but it is not yet safe to execute. The next revision should correct the artifact contract, redesign schema identity, align the page helper with the real Docling structure, and add an explicit automated verification plan. Once those are fixed, this should be reassessed quickly rather than rewritten from scratch.
