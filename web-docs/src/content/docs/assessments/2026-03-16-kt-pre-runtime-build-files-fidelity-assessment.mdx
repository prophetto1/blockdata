# KT Pre-Runtime Build Files Fidelity Assessment

- Source path: `kt/blockdata/`
- Date reviewed: `2026-03-16`
- Reviewer: `Codex`
- Reference style: user-provided runtime-file fidelity assessment from this session
- Kestra core repo used for comparison: `/home/jon/kestra-repos/kestra`
- Kestra MongoDB plugin repo used for comparison: `/home/jon/blockdata/kestra.plugins/plugin-mongodb`
- Verdict: `Conditional Pass`

## Scope

This assessment covers the important `kt` files that existed before the new worker/runtime additions and that were not already the focus of the runtime-file review:

- `core/models/property.py`
- `core/models/tasks/common.py`
- `runtime/registry.py`
- `runtime/connection.py`
- `runtime/routes.py`
- `main.py`
- `connectors/mongodb/*`

It intentionally does not re-score the new worker/runtime files already covered in the runtime assessment.

## Executive Summary

The pre-runtime picture is stronger than the new infrastructure layer.

The MongoDB plugin family is mostly good. The CRUD and query leaves are generally high fidelity to the real `plugin-mongodb` Java sources. The shared plugin substrate is medium fidelity: structurally recognizable, but simplified around JSONL, local storage, and direct Python execution instead of Kestra's internal storage, Ion serde, and richer plugin metadata.

The weakest pieces are not the MongoDB leaves. The weakest pieces are the non-plugin foundation and host files:

- `property.py` is a very thin value wrapper compared with Kestra's real `Property<T>` system.
- `registry.py` is a static lookup table, not a real analog of `PluginRegistry` plus `PluginScanner`.
- `connection.py`, `routes.py`, and `main.py` are BlockData host shims, not faithful translations of Kestra classes.

So the correct conclusion is:

- the MongoDB task family is mostly a credible translation
- the generic plugin/runtime substrate that surrounds it is still compressed
- the host layer should be described as BlockData-specific glue, not as one-to-one Kestra translation

## File-By-File Assessment

### 1. `property.py` — LOW FIDELITY

Kestra `Property<T>` is a full rendered-property system with:

- value vs expression modes
- serializer and deserializer support
- typed conversion via `as(...)`
- list conversion via `asList(...)`
- cache skipping via `skipCache()`

Python `Property` is just:

- `value`
- `of_value()`

What is missing:

- no expression mode
- no skip-cache behavior
- no serialization contract
- no type coercion logic on the class itself
- no list-aware rendering path

Verdict:

This is not a faithful translation of Kestra `Property.java`. It is a deliberately tiny wrapper because most rendering logic was moved into `RunContext`. That is workable for the current prototype, but it is a major reduction in scope.

### 2. `common.py` — HIGH FIDELITY

Kestra `FetchType` has:

- `STORE`
- `FETCH`
- `FETCH_ONE`
- `NONE`

Python `FetchType` has the same four enum values with the same semantics.

Verdict:

This is a clean translation.

### 3. `registry.py` — LOW FIDELITY

Kestra splits this concern across `PluginRegistry` and `PluginScanner`:

- classpath scanning
- metadata loading
- aliases
- external plugin registration
- version support
- typed plugin categories

Python `registry.py` is a hardcoded map from task type to class plus a function-name map.

What is preserved:

- task type to plugin-class resolution
- a minimal list surface for the host API

What is missing:

- scanning
- metadata
- aliases beyond one special-case insert-one alias
- versioning
- plugin categories
- external plugin registration lifecycle

Verdict:

This is enough for a contained prototype, but it is not faithful to Kestra's real registry and scanner architecture.

### 4. `connection.py` — N/A

There is no clean one-class Kestra equivalent for this file.

`connection.py` is a BlockData host shim that:

- tries `app.infra.connection.resolve_connection_sync`
- falls back to environment variables

This is closer to platform-specific secret and credential wiring than to any single Kestra class.

Verdict:

Do not score this as a translation. Score it as BlockData-specific host glue.

### 5. `routes.py` — LOW FIDELITY

The nearest Kestra analog is `ExecutionController`, but that controller is a full execution API:

- execution search and retrieval
- file handling
- previews
- webhook execution paths
- queue interactions
- flow and execution services

`routes.py` is a narrow FastAPI bridge:

- `POST /{function_name}`
- `GET /`

Verdict:

This is not a faithful translation of Kestra's web layer. It is a deliberate host-side shim for plugin execution. That is acceptable, but it should be described that way.

### 6. `main.py` — N/A

There is no single Kestra class that maps to `main.py`.

This file is just the Python app bootstrap:

- create `FastAPI`
- register routes
- log registered plugins
- run `uvicorn`

Verdict:

This is a host bootstrap file, not a translation.

### 7. `abstract_task.py` — HIGH FIDELITY

Kestra `AbstractTask.java` owns:

- `connection`
- `database`
- `collection`
- collection resolution from rendered properties

Python `AbstractTask` preserves that shape almost exactly.

Minor gaps:

- no typed overload for collection class selection
- no Kestra-specific exception type

Verdict:

This is a strong translation.

### 8. `abstract_load.py` — MEDIUM FIDELITY

The structure is clearly derived from Kestra `AbstractLoad`:

- `from`
- `chunk`
- `source(...)`
- batch processing
- metrics
- summary output

Major differences:

- Kestra uses `FileSerde`, Flux, and `collection.bulkWrite(...)`
- Python uses synchronous iteration and manually applies custom write models one by one
- Kestra metrics include database and collection tags
- Python metrics are plain key/value counters

Verdict:

The file preserves the family shape well, but the execution model is substantially simplified.

### 9. `mongodb_connection.py` — MEDIUM FIDELITY

Kestra `MongoDbConnection` is simple:

- render `uri`
- create `MongoClient`

Python preserves that, but adds:

- `connection_id`
- host credential lookup through `runtime.connection`
- `MongoClientFactory` indirection for tests

Verdict:

The core behavior is faithful. The extra `connection_id` path is a BlockData adaptation, not a Kestra translation.

### 10. `mongodb_service.py` — HIGH FIDELITY

Kestra `MongoDbService` has two core jobs:

- convert rendered values into BSON documents
- map BSON values back into plain output shapes

Python preserves both jobs and maps the main MongoDB primitives correctly:

- `ObjectId`
- `Binary`
- `Decimal128`
- `Int64`
- nested documents
- arrays

Minor differences:

- Python returns raw values for more unsupported cases instead of failing as aggressively as Java
- Python preserves `_id` ordering explicitly
- timestamp handling is adapted to Python datetime behavior

Verdict:

This is one of the stronger translations in the repo.

### 11. `find.py` — HIGH FIDELITY

Kestra `Find` supports:

- filter
- projection
- sort
- limit
- skip
- optional stored output

Python preserves all of those.

Notable differences:

- stored output is JSONL via local storage, not Ion via Kestra internal storage
- metrics are simplified
- logging is omitted

Verdict:

High fidelity on task behavior. The main differences are storage format and runtime substrate, not leaf logic.

### 12. `aggregate.py` — MEDIUM FIDELITY

The Python file preserves the main surface:

- `pipeline`
- `allowDiskUse`
- `maxTimeMs`
- `batchSize`
- `store`

The main gap is that two of those preserved fields are not actually applied:

- `max_time_ms`
- `batch_size`

Kestra applies both to the Mongo aggregation cursor. Python currently defines them but does not enforce them.

Verdict:

Directionally correct, but not fully faithful in execution behavior.

### 13. `insert_one.py` — HIGH FIDELITY

This is a straightforward translation:

- render document
- insert one
- emit inserted id and acknowledgment
- emit insert metric

Verdict:

Clean and close to the Java source.

### 14. `update.py` — HIGH FIDELITY

Python preserves:

- `REPLACE_ONE`
- `UPDATE_ONE`
- `UPDATE_MANY`
- filter rendering
- document rendering
- output shape with `upserted_id`, `matched_count`, and `modified_count`

Minor omissions:

- no logger usage
- metric tagging is simplified

Verdict:

This is a strong translation.

### 15. `delete.py` — HIGH FIDELITY

Python preserves:

- `DELETE_ONE`
- `DELETE_MANY`
- filter rendering
- acknowledgment output
- deleted-count output

Verdict:

This is a strong translation.

### 16. `load.py` — MEDIUM FIDELITY

The translation preserves:

- `idKey`
- `removeIdKey`
- conversion of selected source field into `_id`
- inheritance from `AbstractLoad`

The main simplification is input handling:

- Kestra uses `FileSerde.readAll(...)`
- Python reads line-by-line JSON directly with `json.loads`

Verdict:

The task behavior is right, but the storage and serialization path is simplified.

### 17. `bulk.py` — MEDIUM FIDELITY

The high-level shape is preserved:

- parse NDJSON bulk operations
- map each row to insert/update/delete models
- delegate to `AbstractLoad`

Missing relative to Kestra:

- replace options beyond `upsert`
- update options such as `arrayFilters`
- `bypassDocumentValidation`
- `collation`

Verdict:

This is a credible first translation, but it does not yet preserve the full option surface of the real plugin.

### 18. `trigger.py` — LOW FIDELITY

This is the biggest gap in the plugin family.

Kestra `Trigger.java` is a real trigger type:

- extends `AbstractTrigger`
- implements `PollingTriggerInterface`
- evaluates with `ConditionContext` and `TriggerContext`
- returns `Optional<Execution>`
- delegates to `TriggerService.generateExecution(...)`

Python `Trigger` is just:

- a dataclass with interval and query fields
- `Find` reuse
- `evaluate(run_context) -> dict | None`

Verdict:

The query reuse is sensible, but this is not a faithful translation of Kestra's trigger model. It is a simplified adapter.

### 19. `write_models.py` — N/A

Kestra does not have a corresponding source file because it uses MongoDB's native `WriteModel` classes directly.

This file exists to:

- decouple tests from PyMongo internals
- make the Python `AbstractLoad` path easier to unit test

Verdict:

Treat this as a BlockData support file, not a translation.

### 20. `package_info.py` — LOW FIDELITY

Kestra `package-info.java` carries structured plugin subgroup metadata.

Python `package_info.py` is just a descriptive docstring.

Verdict:

Useful documentation, but not a faithful metadata translation.

## Overall Assessment

The pre-runtime build files split into three tiers.

### High fidelity

These are the strongest files:

- `common.py`
- `abstract_task.py`
- `mongodb_service.py`
- `find.py`
- `insert_one.py`
- `update.py`
- `delete.py`

These preserve the shape and behavior of the original Java sources well enough that the MongoDB family is recognizable and useful.

### Medium fidelity

These preserve the right architecture, but compress the execution and storage model:

- `abstract_load.py`
- `mongodb_connection.py`
- `aggregate.py`
- `load.py`
- `bulk.py`

These are acceptable for the current MongoDB-first prototype, but they are not yet fully faithful.

### Low fidelity

These are real translation attempts, but they are materially compressed or structurally divergent from the Kestra originals:

- `property.py`
- `registry.py`
- `routes.py`
- `trigger.py`
- `package_info.py`

These should not be presented as strong Kestra translations yet.

### Not translations

These files are BlockData-specific host glue rather than one-to-one Kestra translations:

- `connection.py`
- `main.py`

These are valid supporting files, but they should be described as platform/bootstrap code, not fidelity work.

## Gaps That Matter Next

1. `property.py` is too thin if `kt` is going to become a true runtime/core substrate rather than a MongoDB-first prototype.
2. `registry.py` is too static if more plugin families are expected.
3. `aggregate.py` should either apply `max_time_ms` and `batch_size` or document them as deferred.
4. `bulk.py` and `load.py` need a richer serde and option path if the goal is closer plugin fidelity.
5. `trigger.py` should stay clearly marked as a simplified adapter until trigger worker infrastructure exists.

## Bottom Line

The MongoDB connector family is in better shape than the surrounding substrate.

If the question is, "Did `kt` translate the MongoDB plugin family credibly?" the answer is mostly yes.

If the question is, "Did `kt` translate the generic Kestra plugin and host architecture faithfully before the new runtime files?" the answer is no. Those parts are still compressed, stubbed, or BlockData-specific.

That means the correct positioning remains:

- present the MongoDB leaf tasks as mostly high-fidelity translation work
- present the shared substrate as partial and intentionally simplified
- do not over-claim the host layer as a one-to-one Kestra translation
