## Goal

Replace the monolithic `ServicesPanel` with a scalable master-detail admin experience that stays responsive at 250+ services. Keep the existing Settings shell and ScrollArea, but split Services into a virtualized list and an on-demand detail editor.

---

## Why Option C (Master-Detail)

| Option | Pros | Cons |
|---|---|---|
| **A: CSS/containment tweaks** | Fastest patch | Doesn't fix core scalability; still mounts thousands of inputs |
| **B: Accordion-only cards** | Lower DOM count, minimal arch change | Preserves monolith and full refetch |
| **C: Master-detail + targeted loading** | Fixes rendering, UX clarity, and data-flow scalability | Multi-file refactor |

**Recommendation:** Option C, with one guardrail from B (single expanded detail panel only).

---

## Architecture

```
┌─────────────────────────────────┐
│  Settings shell (SettingsLayout) │
│  ┌───────────┬─────────────────┐ │
│  │ Service   │ Service Detail  │ │
│  │ List Grid │ Panel           │ │
│  │ (master)  │ (detail)        │ │
│  │           │                 │ │
│  │ RDG       │ Read-only →     │ │
│  │ paginated │ Edit mode       │ │
│  │           │                 │ │
│  │           │ Functions table │ │
│  │           │ (selected svc)  │ │
│  └───────────┴─────────────────┘ │
└─────────────────────────────────┘
```

**Data flow:** `useServicesRegistry` hook manages all state. Initial load fetches `view=services` (summary). Selecting a service fetches `view=service-detail` (one service + its functions). Mutations patch local state and refresh only relevant slices.

## Tasks

### Task 1: Scalable Read Modes on `admin-services` Endpoint

**Files:** `supabase/functions/admin-services/index.ts`

Add two GET view modes:

| View | Params | Returns |
|---|---|---|
| `view=services` | none | Service rows + `function_count` per service (no function payload) |
| `view=service-detail` | `service_id` | One service + only its functions |

Keep current full payload as fallback for compatibility.

---

### Task 2: Extract Services Data Layer

**Files:** `web/src/pages/settings/useServicesRegistry.ts`, `services-registry.types.ts`

Hook returning:
- `serviceSummaries`, `selectedServiceId`, `selectedService`, `selectedFunctions`
- `selectService`, `refreshSummaries`, `refreshSelected`
- Mutation actions with per-entity pending maps (not one global key)

Wire `ServicesPanel` to hook without changing UI.

---

### Task 3: Service List Grid (Master Pane)

**Files:** `web/src/pages/settings/ServicesListGrid.tsx`

`react-data-grid` with columns: Name, Type, Base URL, Health, Enabled, Function Count, Updated.

- Client-side pagination
- Row selection controls detail pane
- No inline editors

---

### Task 4: Detail Pane with Intentional Editing

**Files:** `web/src/pages/settings/ServiceDetailPanel.tsx`, `ServiceFunctionsTable.tsx`

- Default read-only summary
- Explicit Edit button to enter edit mode
- Function table mounted only for selected service
- Confirmation dialog for destructive actions
- Lazy Monaco mounting (edit mode only)

---

### Task 5: Realtime + Refresh Hardening

- `service_registry` change → refresh summary list only
- `service_functions` change → refresh selected detail only if affected
- Debounce multiple events (200-300ms)

---

### Task 6: Remove Legacy Monolith

Delete from `ServicesPanel`:
- Global `serviceSavingKey`
- Bulk draft maps keyed for all rows
- Full-list inline edit table rendering path

---

## Definition of Done

- Services tab responsive with 250+ services
- Only one service detail mounted at a time
- Mutations don't trigger unconditional full refetch
- Operators can scan list quickly and edit intentionally
