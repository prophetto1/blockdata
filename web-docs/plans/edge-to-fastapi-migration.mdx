## Decision

Replace all Supabase edge functions (Deno/TypeScript) with a single FastAPI service (`blockdata-api`) running locally alongside the existing `pipeline-worker` and `conversion-service`.

**Why:**
- Edge functions hit memory limits (WORKER_LIMIT / status 546) on large payloads
- No local debugging ‚Äî deploy-to-test cycle is slow
- Deno runtime quirks (ES256 JWT incompatibility forced `verify_jwt=false` on multiple functions)
- Python ecosystem is richer for the data/ML work blockdata does
- Three FastAPI services already exist locally; adding a fourth is consistent

**What stays in Supabase:** Auth (Supabase Auth tokens), the Postgres database, Storage buckets. FastAPI talks to the same Supabase Postgres via direct connection string or `supabase-py`.

## Shared Modules to Port

These TypeScript modules in `supabase/functions/_shared/` must be rewritten in Python:

| Module | Purpose | Python Equivalent |
|--------|---------|-------------------|
| `cors.ts` | CORS headers | FastAPI CORSMiddleware (built-in) |
| `superuser.ts` | Superuser email allowlist gate | Python middleware or dependency |
| `supabase.ts` | Admin/user Supabase client, requireUserId | `supabase-py` or `asyncpg` + auth middleware |
| `env.ts` | Environment variable helpers | `os.environ` / `pydantic-settings` |
| `vertex_auth.ts` | GCP service account JWT ‚Üí OAuth2 token | `google-auth` library |
| `vertex_claude.ts` | Vertex AI Claude transport | `anthropic[vertex]` SDK or `httpx` |
| `vertex_claude_stream.ts` | Streaming variant | Same SDK, streaming mode |
| `api_key_crypto.ts` | XChaCha20-Poly1305 encryption | `pynacl` or `cryptography` library |
| `admin_policy.ts` | Runtime policy resolution | Direct SQL query |
| `hash.ts` | SHA-256 hashing | `hashlib` (stdlib) |
| `sanitize.ts` | Input sanitization | Python equivalent |
| `user_schema.ts` | Schema validation helpers | Python equivalent |

**Not ported** (handled by conversion-service already): `citations.ts`, `docling.ts`, `markdown.ts`, `pandoc.ts`, `representation.ts`, `conversion-ack-timeout.ts`

## Auth Migration

| Current Pattern | Used By | FastAPI Approach |
|----------------|---------|-----------------|
| `requireSuperuser` (JWT + email allowlist) | admin-config, admin-database-browser, admin-integration-catalog, admin-services | Dependency that decodes Supabase JWT, checks email against `SUPERUSER_EMAIL_ALLOWLIST` env var |
| `requireUserId` (JWT ‚Üí user_id) | 14 user-facing functions | Dependency that decodes Supabase JWT, extracts `sub` claim as user_id |
| `X-Conversion-Service-Key` (shared secret) | conversion-complete | Header check against env var |

Supabase Auth stays. The frontend still gets JWTs from Supabase. FastAPI validates those same JWTs using the Supabase JWT secret (available as `SUPABASE_JWT_SECRET` env var). No auth system change ‚Äî just moving the verification from Deno to Python.

## Frontend Migration

Every `edgeFetch('function-name')` call in `web/src/` switches to `fetch('http://localhost:8001/...')`.

The response shapes stay identical ‚Äî same JSON structure, same field names. This means UI components don't change, only the transport layer in the API client files.

**API client files to update:**
- `integration-catalog.api.ts` ‚Äî `edgeFetch('admin-integration-catalog')`
- `services-panel.api.ts` ‚Äî `edgeFetch('admin-services')`
- Other API files in `web/src/` that call `edgeFetch()`

A single `apiBase` constant or environment variable controls the base URL, so the switch is one line per API client file.

## Migration Order

| Phase | Functions | Rationale |
|-------|-----------|-----------|
| **Phase 1** | admin-config, admin-database-browser, admin-integration-catalog, admin-services | Admin-only, superuser auth, lowest user impact, proves the pattern |
| **Phase 2** | schemas, runs, flows, user-api-keys, provider-connections, agent-config, upload-policy, export-jsonl, test-api-key | User-facing CRUD, standard auth, no external service calls |
| **Phase 3** | ingest, trigger-parse, conversion-complete, generate-citations | Pipeline orchestration, callback handling, conversion-service integration |
| **Phase 4** | worker | The big one ‚Äî 1,442 lines, Vertex AI integration, prompt caching, batch processing |
| **Phase 5** | assistant-chat | SSE streaming, Vertex Claude, thread management |

Each phase is independently deployable. Frontend switches one router at a time. Old edge functions stay deployed until their FastAPI replacement is verified.

## Open Questions

1. **Port or connection string?** Should blockdata-api use `supabase-py` (REST, same as edge functions use the JS client) or `asyncpg` (direct Postgres connection, faster, no REST overhead)? Recommendation: `asyncpg` for reads, `supabase-py` for auth-related operations.

2. **Crypto compatibility.** The edge functions encrypt API keys with XChaCha20-Poly1305 via Deno's `tweetnacl`. The Python side must use the same algorithm and key to decrypt existing encrypted keys. Need to verify `pynacl` produces identical ciphertext.

3. **Vertex AI SDK.** The edge functions use raw `fetch()` to Vertex AI. Python has `anthropic[vertex]` SDK which is cleaner. But need to verify prompt caching and streaming work identically.

4. **Conversion-complete callback.** Currently conversion-service POSTs back to the edge function. After migration, it POSTs to `http://localhost:8001/parse/conversion-complete` instead. Both services run locally so this is a config change in conversion-service.

5. **Edge function retirement.** After all phases complete, do we delete the edge functions from the repo or keep them as reference? Recommendation: archive to a `_deprecated/` directory, delete after 30 days of stable FastAPI operation.
