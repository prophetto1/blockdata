## Decision

Replace all Supabase edge functions (Deno/TypeScript) with a single FastAPI service (`blockdata-api`) running locally alongside the existing `pipeline-worker` and `conversion-service`.

**Why:**
- Edge functions hit memory limits (WORKER_LIMIT / status 546) on large payloads
- No local debugging — deploy-to-test cycle is slow
- Deno runtime quirks (ES256 JWT incompatibility forced `verify_jwt=false` on multiple functions)
- Python ecosystem is richer for the data/ML work blockdata does
- Three FastAPI services already exist locally; adding a fourth is consistent

**What stays in Supabase:** Auth (Supabase Auth tokens), the Postgres database, Storage buckets. FastAPI talks to the same Supabase Postgres via direct connection string or `supabase-py`.

                         ▼

**blockdata-api** owns all web app API endpoints. It does NOT execute plugins (that's pipeline-worker) or parse documents (that's conversion-service). It is the API gateway between the frontend and the database.

## Auth Migration

| Current Pattern | Used By | FastAPI Approach |
|----------------|---------|-----------------|
| `requireSuperuser` (JWT + email allowlist) | admin-config, admin-database-browser, admin-integration-catalog, admin-services | Dependency that decodes Supabase JWT, checks email against `SUPERUSER_EMAIL_ALLOWLIST` env var |
| `requireUserId` (JWT → user_id) | 14 user-facing functions | Dependency that decodes Supabase JWT, extracts `sub` claim as user_id |
| `X-Conversion-Service-Key` (shared secret) | conversion-complete | Header check against env var |

Supabase Auth stays. The frontend still gets JWTs from Supabase. FastAPI validates those same JWTs using the Supabase JWT secret (available as `SUPABASE_JWT_SECRET` env var). No auth system change — just moving the verification from Deno to Python.

## Frontend Migration

Every `edgeFetch('function-name')` call in `web/src/` switches to `fetch('http://localhost:8001/...')`.

The response shapes stay identical — same JSON structure, same field names. This means UI components don't change, only the transport layer in the API client files.

**API client files to update:**
- `integration-catalog.api.ts` — `edgeFetch('admin-integration-catalog')`
- `services-panel.api.ts` — `edgeFetch('admin-services')`
- Other API files in `web/src/` that call `edgeFetch()`

A single `apiBase` constant or environment variable controls the base URL, so the switch is one line per API client file.

## Migration Order

| Phase | Functions | Rationale |
|-------|-----------|-----------|
| **Phase 1** | admin-config, admin-database-browser, admin-integration-catalog, admin-services | Admin-only, superuser auth, lowest user impact, proves the pattern |
| **Phase 2** | schemas, runs, flows, user-api-keys, provider-connections, agent-config, upload-policy, export-jsonl, test-api-key | User-facing CRUD, standard auth, no external service calls |
| **Phase 3** | ingest, trigger-parse, conversion-complete, generate-citations | Pipeline orchestration, callback handling, conversion-service integration |
| **Phase 4** | worker | The big one — 1,442 lines, Vertex AI integration, prompt caching, batch processing |
| **Phase 5** | assistant-chat | SSE streaming, Vertex Claude, thread management |

Each phase is independently deployable. Frontend switches one router at a time. Old edge functions stay deployed until their FastAPI replacement is verified.

## Design Review

### Service boundary: blockdata-api vs pipeline-worker

Both services talk to Vertex AI — blockdata-api for the worker (extraction) and chat (streaming), pipeline-worker for Kestra plugin execution. Two services with duplicate Vertex AI clients, duplicate auth, and duplicate credential resolution is wasteful.

**Decision needed:** Does pipeline-worker eventually absorb the worker function, or does blockdata-api stay the sole Vertex AI consumer? The cleaner boundary is:
- **blockdata-api** = web API gateway, CRUD, orchestration dispatch
- **pipeline-worker** = all compute (LLM extraction, plugin execution, chat)

This avoids duplicating Vertex AI infra across two services.

### Phase 4 (worker) is 40% of the total work

The worker function is 1,442 lines that evolved through 15 production iterations: adaptive batching, prompt cache management, overflow recovery, Vertex AI streaming, cost tracking, policy snapshot enforcement. Porting this is a rewrite, not a translation. **Phase 4 needs its own sub-plan with an explicit test matrix for each behavior.**

### Database access: pick one pattern

Don't mix `asyncpg` and `supabase-py` in the same service. Two connection pools, two query styles, two error handling paths. Use `asyncpg` exclusively with a connection pool (`asyncpg.create_pool`). Write a thin auth helper that queries the same pool. The edge functions use PostgREST because they have to — this migration removes that constraint.

### Encryption compatibility is a pre-Phase-2 gate

XChaCha20-Poly1305 keys encrypted by Deno's `tweetnacl` must be decryptable by Python's `pynacl`. Same algorithm, but nonce handling and key derivation details can differ between implementations. If decryption fails, existing API keys are unreadable. **Run a compatibility spike before Phase 2, not during it.**

### Connection pooling not addressed

19 edge functions each create ephemeral Supabase clients per invocation. A single FastAPI service with `asyncpg` holds persistent connections. Size the pool for concurrent request load. Not hard, but must be configured explicitly in `db.py`.

### Conversion-complete callback is a deploy sequencing problem

conversion-service POSTs to the edge function URL. When blockdata-api takes over, conversion-service must POST to `http://localhost:8001/parse/conversion-complete` instead. Deploy order matters — if either side switches before the other, callbacks fail. Options:
- Run both endpoints in parallel during transition (edge function forwards to FastAPI)
- Deploy atomically (update conversion-service config and FastAPI endpoint together)

## Execution System

### Design Decisions

**Triggers:** Phase 2. Get manual "user clicks Run" execution working first. Automatic triggers (schedules, webhooks, file watchers, event-driven hooks) come later. Everything built so far — service registry, integration catalog, flow_sources — assumes manual execution as the first mode.

**Flow node references:** Nodes reference `task_class` (e.g. `io.kestra.plugin.dbt.cli.DbtCLI`), not `function_id`. Reasons:
- Kestra's naming system becomes the universal plugin vocabulary across the platform
- `function_id` is a UUID that changes if you re-provision services
- The catalog → service mapping is an admin concern, not a flow author concern
- A flow built against `io.kestra.plugin.dbt.cli.DbtCLI` stays valid even if the backing service is swapped

### Execution Chain (end to end)

User clicks Run → resolution → dispatch → execute → track → return result

| Step | What It Does | Status | Evidence |
|------|-------------|--------|----------|
| 1. Plugin catalog | task_classes with input/output schemas | **Built, 945 items** | `integration_catalog_items` (migration 059) — columns: task_class, task_schema (JSONB), mapped_service_id, mapped_function_id. Kestra sync complete. |
| 2. Service registry | Services with base_url + functions with entrypoint | **Built, seeded** | `service_registry` (6 rows), `service_functions` (31 rows), `service_functions_view` (joined). Migration 050. |
| 3. Mapping | task_class → mapped_function_id → service | **Schema built, 2/945 mapped** | Only Slack (SlackIncomingWebhook, SlackExecution) mapped. Rest unmapped. |
| 4. Resolution | Code that walks task_class → catalog → function → service.base_url + entrypoint | **Missing** | No blockdata-api resolution function exists. pipeline-worker has its own internal `registry.resolve()` but that's local plugin dispatch, not DB-level catalog resolution. |
| 5. Parameter form | UI renders task_schema inputs, user fills values, clicks Execute | **Missing** | ServicesPanel is CRUD-only. No form generation from parameter_schema. |
| 6. Dispatch | HTTP POST to resolved service URL with params + context | **Missing** | No edge function or API endpoint creates service_runs rows or POSTs to backing services. |
| 7. Backing services | Something listening at base_url that executes the task | **Partially built** | pipeline-worker: 13 plugin classes, 26 task_class strings, POST `/execute` endpoint (`services/pipeline-worker/`). dlt/dbt: endpoints seeded in service_functions but no backing FastAPI handlers yet. |
| 8. Run tracking | Create run row, update status, store result/artifacts | **`service_runs` built, `service_run_events` + `service_run_artifacts` do not exist** | `service_runs` (migration 050). `service_run_events` and `service_run_artifacts` migrations exist in code (051) but were never applied to live DB. |
| 9. Flow execution | Walk a DAG of tasks, execute in order, pass outputs downstream | **Missing** | `flow_sources` table (migration 049) stores YAML source + revision. No execution engine. |
| 10. Orchestrator | DAG dependency resolution, fan-out, branching, retry, timeout | **Missing** | This is the hard one. |

### What's Missing (5 items to build)

**1. Resolution function** — `resolve(task_class) → {base_url, entrypoint, http_method, params_schema}`
- Small function in blockdata-api
- Reads `integration_catalog_items` → `service_functions` → `service_registry` in one joined query
- Falls back to pipeline-worker's internal registry for unmapped task_classes

**2. Dispatch + run tracking** — blockdata-api endpoint: `POST /execute`
- Accepts `{task_class, params, project_id?}`
- Calls resolution function
- Creates `service_runs` row (status=pending)
- POSTs to the resolved `base_url + entrypoint`
- Updates row (status=complete/failed), stores output
- Note: `service_run_events` and `service_run_artifacts` tables do not exist yet (migration 051 not applied). Create them if detailed run history / file artifacts are needed.

**3. Parameter form UI** — React component renders `task_schema.properties` as a form
- Text inputs, selects for enums, toggles for booleans, JSON editor for objects
- User fills in values, clicks Execute, submits to dispatch endpoint
- This is the "Run a plugin" surface

**4. Flow orchestrator** — Backend reads flow definition (YAML from flow_sources)
- Topologically sorts tasks by `dependsOn`
- Executes each task via dispatch (sequential or parallel for independent branches)
- Passes outputs from upstream tasks as variable context to downstream tasks
- Handles If/Switch branching by evaluating conditions
- Creates one `service_runs` row per task in the flow

**5. Bulk mapping** — Map the 945 catalog items to services (only 2 mapped today)
- Catalog already populated (945 items with task_schema). Sync is done.
- Pattern-based mapping rules: all `io.kestra.plugin.dbt.*` → dbt service, all `io.kestra.plugin.scripts.*` → pipeline-worker
- Regex rules that bulk-set `mapped_service_id` + `mapped_function_id`, not 945 manual clicks

### Minimum Viable Execution Path

Items 1-3 are the minimum: **resolve → dispatch → parameter form**. That gets "user clicks Run on a single plugin task, sees result" working. Items 4-5 make it usable for flows and real plugin coverage.

| Phase | What | Depends On |

## Missing Sections

### Testing strategy

Each router needs integration tests against a test database. Minimum coverage:
- Auth gates (superuser rejected for user endpoints, user rejected for admin endpoints)
- CRUD round-trips (create → read → update → delete)
- Edge cases from the original TypeScript (overflow recovery, idempotency, hash derivation)

### Rollback plan

If a phase goes wrong, the frontend must switch back to edge functions. During migration:
- Keep edge functions deployed and functional
- Frontend `apiBase` toggle per router (not global) so one broken router doesn't force full rollback
- Edge functions archived only after 30 days of stable FastAPI operation per phase

## Open Questions

1. ~~**Port or connection string?**~~ **Resolved:** Use `asyncpg` exclusively. No `supabase-py` for database operations. Connection pool in `db.py`, thin auth helper querying the same pool.

2. **Crypto compatibility.** Must verify `pynacl` can decrypt existing `tweetnacl`-encrypted API keys before Phase 2 starts. Run a compatibility spike with real encrypted data.

3. **Vertex AI SDK.** The `anthropic[vertex]` SDK is cleaner than raw HTTP. Verify prompt caching and streaming work identically. If blockdata-api keeps the worker, both services need this SDK.

4. **Conversion-complete callback.** Config change in conversion-service. Deploy both sides atomically or run edge function as a forwarding proxy during transition.

5. ~~**Edge function retirement.**~~ **Resolved:** Keep deployed during migration. Archive to `_deprecated/` per phase after 30 days stable. Delete after all phases complete.
