**Goal:** Build a working execution chain where a user clicks Run on any mapped plugin, blockdata resolves the handler, dispatches to the backing service, and tracks the result.

**Architecture:** Resolution function walks `task_class` → `integration_catalog_items` → `service_functions` → `service_registry` to produce `{base_url, entrypoint, http_method, params_schema}`. Dispatch POSTs to the resolved URL, creates a `service_runs` row, and updates it on completion. The parameter form renders `parameter_schema` as a React form.

**Tech Stack:** FastAPI (blockdata-api), React + Ark UI (parameter form), Supabase Postgres (service_runs), pipeline-worker (existing FastAPI on :8000)

---

## Design decisions

### Triggers: Phase 2

Triggers (schedules, webhooks, file watchers, polling) are deferred. Get manual "user clicks Run" execution working first, add automatic event-driven execution later. This matches everything built so far — the service registry, the integration catalog, the flows table — all set up for manual execution before any reactive hooks.

### Flows reference `task_class`

When a user builds a flow on the Eternity Canvas or Transform surface, each node references `task_class` (e.g. `io.kestra.plugin.scripts.python.Script`), not `function_id`. Reasons:

- `task_class` is the stable, universal plugin vocabulary across the platform
- `function_id` is a UUID that can change if services are re-provisioned
- The catalog → service mapping is an admin concern, not a flow author concern
- A flow built against `io.kestra.plugin.dbt.cli.DbtCLI` stays valid even if you swap the backing service
- Resolution at execution time means the orchestrator always uses the latest mapping

## Execution chain

```
User clicks Run
  → 1. Resolution: task_class → catalog → service_functions → service_registry.base_url + entrypoint
  → 2. Dispatch: POST to resolved URL with params + context
  → 3. Track: Create service_runs row (PENDING → RUNNING → SUCCESS/FAILED)
  → 4. Return: Result displayed in UI
```

### Step-by-step with tables

| Step | What it does | Status | Depends on |
|------|-------------|--------|------------|
| 1. Plugin catalog | 945 `task_class` entries with input/output schemas | Built | — |
| 2. Service registry | Services with `base_url` + functions with `entrypoint` | Built | — |
| 3. Mapping | `task_class` → `mapped_function_id` → service | Schema built, data mostly unmapped | Bulk mapping (E6) |
| 4. Resolution | Code that walks `task_class` → catalog → function → `service.base_url` + `entrypoint` | Missing | Mapping data |
| 5. Parameter form | UI renders `task_schema` inputs, user fills values, clicks Execute | Missing | Resolution |
| 6. Dispatch | HTTP POST to resolved service URL with params + context | Missing | Resolution + backing service |
| 7. Backing services | Something listening at `base_url` that executes the task | Partial (pipeline-worker has 13 handlers) | — |
| 8. Run tracking | Create `service_runs` row, update status, store result | Schema built, no code writes to it | Dispatch |
| 9. Flow execution | Walk a DAG of tasks, execute in order, pass outputs downstream | Missing | Dispatch + orchestrator |
| 10. Orchestrator | DAG dependency resolution, fan-out, branching, retry, timeout | Missing | Flow execution |

## What to build (6 items)

### E1. Resolution function

**Scope:** Small. One joined query across 3 tables.

```sql
SELECT
  sr.base_url,
  sf.entrypoint,
  sf.http_method,
  sf.parameter_schema,
  sf.function_name
FROM integration_catalog_items ci
JOIN service_functions sf ON sf.function_id = ci.mapped_function_id
JOIN service_registry sr ON sr.service_id = ci.mapped_service_id
WHERE ci.task_class = $1
  AND sr.enabled = true
  AND sf.enabled = true;
```

**Lives in:** `blockdata-api` as a shared utility or FastAPI dependency.

**Returns:** `{base_url, entrypoint, http_method, parameter_schema, function_name}` or 404 if unmapped/disabled.

### E2. Dispatch + run tracking

**Scope:** One FastAPI endpoint in blockdata-api.

`POST /execute` accepts `{task_class, params, project_id?}`:

1. Call resolution (E1) to get `{base_url, entrypoint, http_method}`
2. Create `service_runs` row with status `PENDING`
3. Update to `RUNNING`, record `started_at`
4. HTTP POST/GET to `base_url + entrypoint` with params
5. Update `service_runs` with `SUCCESS` or `FAILED`, `result`, `error_message`, `completed_at`, `duration_ms`
6. Return result to caller

**Kestra parallel:** `ExecutionController.java` creates an Execution, emits to queue. `DefaultWorker.java` picks up WorkerTask, calls `task.run(runContext)`. blockdata simplifies this to a direct HTTP call — no queue needed for Phase 1.

### E3. Pipeline-worker alignment

**Scope:** Verify pipeline-worker's `/execute` endpoint accepts the same shape blockdata-api will POST.

Pipeline-worker already has:
```
POST /execute { task_type, params, execution_id, task_run_id, variables }
```

blockdata-api dispatch will POST:
```
POST http://localhost:8000/execute { task_type: <task_class>, params: <user_params>, execution_id: <run_id> }
```

**Gap:** Pipeline-worker resolves `task_type` against its internal registry (13 handlers, 26 task_class strings). For unmapped task_classes (e.g. `io.kestra.plugin.slack.SlackIncomingWebhook`), pipeline-worker returns 404. New plugin handlers must be added for each mapped service category.

### E4. Parameter form UI

**Scope:** React component that renders `parameter_schema` (JSON array of `{name, type, required, default, description, values}`) as a form.

**Type dispatch** (same pattern as Kestra's `getTaskComponent.ts`):

| `type` | Component |
|--------|-----------|
| `string` | Text input |
| `string` + `values` | Select (Ark UI) |
| `integer` / `number` | Number input (Ark UI) |
| `boolean` | Switch (Ark UI) |
| `object` | JSON editor (Monaco or textarea) |
| `array` | Tag input or JSON editor |

User fills values → clicks Execute → `POST /execute { task_class, params }`.

### E5. Bulk mapping

**Scope:** Pattern-based rules to map catalog items to services in bulk, not 945 manual clicks.

**Strategy:** Match `task_class` prefix → service + function template.

| Pattern | Service | Notes |
|---------|---------|-------|
| `io.kestra.plugin.scripts.python.*` | pipeline-worker | Already has handler |
| `io.kestra.plugin.scripts.shell.*` | pipeline-worker | Already has handler |
| `io.kestra.plugin.scripts.node.*` | pipeline-worker | Already has handler |
| `io.kestra.plugin.core.http.*` | pipeline-worker | Already has handler |
| `io.kestra.plugin.core.log.*` | pipeline-worker | Already has handler |
| `io.kestra.plugin.core.flow.*` | pipeline-worker | Already has handler |
| `io.kestra.plugin.dbt.*` | transform-runner | 13 items |
| `io.kestra.plugin.dlt.*` | load-runner | 2 items |
| `io.kestra.plugin.slack.*` | slack | Done (2 items) |

**Implementation:** SQL migration or admin endpoint that applies these rules. Creates `service_functions` rows where needed and sets `mapped_service_id` + `mapped_function_id` on matching catalog items.

### E6. Flow orchestrator (Phase 2)

**Scope:** Reads a flow definition from `flow_sources`, topologically sorts tasks, executes each via dispatch (E2), passes outputs downstream.

**Deferred.** Depends on E1-E5 being stable. Start with sequential execution, add DAG resolution later.

**Kestra parallel:** `JdbcExecutor.java` polls queues, `FlowableUtils.resolveDagNexts()` checks `dependsOn` for each task. blockdata can start much simpler — linear execution first, then add `dependsOn` support.

## Build order

| Phase | Items | What it proves |
|-------|-------|---------------|
| **E1-E3** | Resolution + dispatch + worker alignment | A user can execute a mapped plugin and get a result back. Minimum viable execution. |
| **E4** | Parameter form | A user can fill in params via UI and execute. First real user-facing surface. |
| **E5** | Bulk mapping | All pipeline-worker-compatible plugins (26 task_classes) become executable without manual mapping. |
| **E6** | Flow orchestrator | Multi-step flows execute automatically. Phase 2. |

E1-E3 are the minimum viable execution path. E4 makes it usable. E5 makes it scalable. E6 makes it powerful.

## Missing database tables

Two tables referenced in earlier discussions do not exist and should be created if needed:

| Table | Purpose | When needed |
|-------|---------|-------------|
| `service_run_events` | Fine-grained event log per run (started, step_completed, failed, retried) | E2 if we want detailed run history beyond the `service_runs` row |
| `service_run_artifacts` | Output files/data produced by a run (stored in Supabase Storage, referenced by run_id) | E2 if plugins produce file artifacts |

Both are optional for E1-E3. `service_runs` alone is sufficient for basic status tracking.

## Open questions

1. **blockdata-api port.** The edge-to-fastapi-migration plan assigns `:8001`. Confirm this doesn't conflict with anything on the dev machine.

2. **Pipeline-worker deployment.** Currently exists as code at `services/pipeline-worker/` but is it actually running? If not, E3 requires starting it.

3. **Auth on dispatch.** When blockdata-api POSTs to pipeline-worker, does it need auth? Both run locally — shared secret header or no auth?

4. **Async execution.** E2 describes synchronous dispatch (wait for result). For long-running plugins (dbt build, LLM extraction), should dispatch be async (return run_id immediately, poll for result)?

5. **Plugin handler gap.** Pipeline-worker has 13 handlers covering 26 task_classes. The catalog has 945 items. As more categories get mapped (E5), new handlers must be written. What's the priority order after scripts/core/http?
