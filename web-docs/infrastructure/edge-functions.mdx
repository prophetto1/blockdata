## Auth Model

All functions disable the Supabase gateway JWT check (`verify_jwt = false` in `config.toml`) due to ES256 JWT incompatibility. Auth is enforced internally:

| Method | Used by |
|---|---|
| `requireUserId(req)` | User-facing functions: `schemas`, `runs`, `ingest`, `flows`, `export-jsonl`, `upload-policy`, `user-api-keys`, `agent-config`, `provider-connections`, `assistant-chat`, `generate-citations` |
| `requireSuperuser(req)` | `admin-config`, `admin-database-browser`, `admin-integration-catalog`, `admin-services` |
| `X-Conversion-Service-Key` header | `conversion-complete` (shared secret with Python conversion service) |
| Provider-specific validation | `test-api-key` |

Superuser gated by `SUPERUSER_EMAIL_ALLOWLIST` secret (`jondev717@gmail.com`).

## Deployment Pipeline

### GitHub Actions (current)

```yaml
on:
  push:
    branches: [master]
    paths: ['supabase/functions/**']
  workflow_dispatch:
```

Deploys all functions atomically via `supabase functions deploy --use-api`.

**Required GitHub secrets:** `SUPABASE_PROJECT_ID` (`dbdzzhshmigewyprahej`), `SUPABASE_ACCESS_TOKEN`.

### Local development

```bash
./scripts/serve-edge.sh <function-name>          # port 8099
./scripts/serve-edge.sh <function-name> 9000     # custom port
```

Loads `.env` from repo root, runs with `deno run --allow-net --allow-env --allow-read`.

## Code Patterns

### Pattern A — Bare `Deno.serve()` (13 functions)

```typescript
Deno.serve(async (req) => {
  const preflight = corsPreflight(req);
  if (preflight) return preflight;
  // handler body
});
```

`admin-config`, `assistant-chat`, `conversion-complete`, `export-jsonl`, `generate-citations`, `ingest`, `runs`, `schemas`, `test-api-key`, `trigger-parse`, `upload-policy`, `user-api-keys`, `worker`.

### Pattern B — Exported handler + `import.meta.main` (6 functions)

```typescript
export async function handleFlowsRequest(req: Request): Promise<Response> {
  // handler body
}

if (import.meta.main) {
  Deno.serve((req) => handleFlowsRequest(req));
}
```

`admin-database-browser`, `admin-integration-catalog`, `admin-services`, `agent-config`, `flows`, `provider-connections`.

Pattern B allows the handler to be imported by a router without triggering `Deno.serve()`. Both work identically on Supabase (runs file as entrypoint → `import.meta.main` is `true`).

## Planned: Self-Hosted Deno Server

Run all 19 functions on the Ubuntu box as a single Deno process, alongside existing Python services. Supabase-hosted functions stay active.

### Steps

1. **Refactor Pattern A → B.** Extract 13 bare `Deno.serve()` handlers into named exports with `import.meta.main` guards. Backward-compatible with Supabase.

2. **Router entrypoint.** `supabase/functions/self-hosted/main.ts` — single `Deno.serve()` importing all 19 handlers, dispatching by URL path (`/functions/v1/{name}` or `/{name}`). Health check at `/health`.

3. **Docker.** `denoland/deno:2.1.4`, copies `supabase/functions/`, pre-caches `esm.sh` deps. Port 9000.

4. **Docker Compose.** Add `edge-functions` service on Ubuntu box. Health check via `/health`, `restart: unless-stopped`.

5. **GitHub Actions deploy.** New workflow: SSH into Ubuntu → `git pull` → rebuild Docker → restart → verify health. Same trigger path as Supabase workflow.

6. **Nginx reverse proxy.** Upstream at `localhost:9000`. `proxy_buffering off` (SSE streaming for `assistant-chat`), `proxy_read_timeout 300s` (long worker runs).

### Considerations

- **Vertex AI token caching** benefits from shared process — one cached GCP token across all functions.
- **`ingest` sub-modules** use relative imports (`./types.ts`) — resolve correctly since router imports from `ingest/index.ts`.
- **`flows` sub-path routing** uses `indexOf("flows")` — prefix-agnostic, works unchanged.
- **No external npm deps** besides `@supabase/supabase-js@2` via `esm.sh`.
- **New GitHub secrets needed:** `UBUNTU_SSH_PRIVATE_KEY`, `UBUNTU_USER`, `UBUNTU_HOST`.
