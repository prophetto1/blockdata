## 0) Authority, scope boundaries, and conflict policy

### 0.1 Canonical authority hierarchy (when docs conflict)
Per `legal-10/docs/_INDEX.md`:
1. `legal-10/docs/_essentials/mvp/M1-buildtime-packaging-sealing-dev-brief.md` (M1) — highest authority for **bundle layout, schemas, and sealing** for the 3-step MVP.

3. `legal-10/docs/fdq/*.md` and `legal-10/docs/fdq/post/*.md` — authoritative for **step prompts, output contracts, and scoring**.
4. Everything else under `legal-10/docs/**` — supplementary, historical, or forward-looking (10-step).

### 0.2 Explicit conflicts identified (and resolved for MVP)
1. **Sealing format conflict**
   - M1 requires a detached `signature.json` at the bundle root, alongside `manifest.json`.
   - `legal-10/docs/build-pipeline/sealed-evaluation-units-security.md` describes an embedded signature approach.
   - **Resolution for 3-step MVP:** M1 is canonical: bundle root MUST contain both `manifest.json` and `signature.json`.
2. **Citation integrity “open-book” validity scope**
   - Older docs imply open-book citations must be valid against anchor inventory only.
   - MVP + FDQ docs for open-book IRAC require citations valid if in **anchor inventory OR research pack subset**.
   - **Resolution for 3-step MVP:** open-book validity uses `anchor_inventory_full ∪ rp_subset`.

### 0.3 Not in scope of the 3-step MVP
The full “10-step chain” (steps 02–08, j10, etc.) is documented under `legal-10/docs/10-step-chain/` and `legal-10/docs/fdq/02–08*.md`, but is **out of scope** for 3-step MVP acceptance, except where the 3-step design explicitly preserves forward compatibility.

## 1) Glossary (normative terms)

- **Benchmark bundle (released bundle):** A sealed directory containing `benchmark/`, `eus/`, `manifest.json`, `signature.json`. This is the “product artifact” of build time and the input to runtime.
- **Benchmark packet:** The content under `benchmark/` (plan, step definitions, judge prompts, system message).
- **EU (Evaluation Unit):** One evaluation instance (one anchor opinion) packaged as `eus/<eu_id>/p1.json`, `p2.json`, `ground_truth.json`.
- **RP (Research Pack):** Build-time intermediate product assembled from datasets; in the released bundle the RP is shipped **as `p2.json` inside each EU**, not as a separate bundle folder.
- **Evaluated model:** The model being scored on d1/d2/j3.
- **Judge model:** A separate model used to grade the d2/j3 IRAC pair via rubric.
- **Candidate state:** The carry-forward JSON object derived from model outputs only; ground truth, scores, and judge artifacts are forbidden.
- **Replay_Minimal:** Session strategy where each step is a fresh model call with no accumulated chat history (hard cut); step-to-step state is carried only through `candidate_state.json` content and admitted payloads.
- **Admitted payloads:** The subset of EU payload files visible to the evaluated model at a step boundary, controlled by `plan.json` `inject_payloads`.
- **No-leak invariant:** The evaluated model must never see `ground_truth.json`, judge prompts, future steps, or unadmitted payloads.

Normative keywords: **MUST**, **SHOULD**, **MAY**.

## 2) Desired end-result product (what “done” looks like)

### 2.1 Build-time deliverable: sealed benchmark bundle
A developer can run build-time CLIs to produce a directory with the exact bundle layout required by M1:

```
<bundle_root>/
  benchmark/
    benchmark.json
    plan.json
    model_steps/
      d1.json
      d2.json
      j3.json
    judge_prompts/
      irac_mee_pair_v1.json
  eus/
    <eu_id>/
      p1.json
      p2.json
      ground_truth.json
    ... (200 total)
  manifest.json
  signature.json
```

This bundle is cryptographically sealed:
- `manifest.json` inventories all required files with SHA-256 and byte counts.
- `signature.json` is an Ed25519 detached signature over `manifest.json` bytes.

### 2.2 Runtime deliverable: runner that produces auditable run artifacts
Given a sealed bundle and model credentials, the runner executes the 3-step chain per EU and writes per-run artifacts:
- `run.jsonl` (append-only event log of model/judge/post-chain steps)
- `audit_log.jsonl` (hash proofs: staged file hashes, message hash, response hash, etc.)
- `candidate_state.json` (sanitized carry-forward outputs)
- `summary.json` (scores + rollups)
- `run_manifest.json` (provenance + hashes of run outputs)
- optional: `trace.jsonl` for structured trace events (future / non-blocking for MVP unless required by acceptance criteria)

## 3) Artifact specifications (RP, EU, benchmark packet, bundle sealing)

### 3.1 EU package (`eus/<eu_id>/…`) — what the runner consumes

#### 3.1.1 `p1.json` (“anchor payload”) — candidate visible
Representative (see `legal-10/datasets/eus/legal10_3step_v1/eus/eu__1826-018/p1.json`):
- Top-level:
  - `type: "anchor"`
  - `payload_id: "p1"`
  - `payload_version` (semver string)
  - `candidate_visible: true`
  - `content.anchor`:
    - `caseId` (string, e.g. `"1826-018"`)
    - `usCite` (string, e.g. `"24 U.S. 380"`)
    - `caseName` (string)
    - `term` (int)
    - `text` (string; full anchor opinion text; never trimmed in MVP)
    - `char_count` (int)
  - `metadata` (build-time provenance and helper indices; candidate-visible unless excluded by policy):
    - `citations[]` roster items with `inventory_normalized_cite`, `cite_offset`, ids, etc.
    - `rp_id` (string) linking to build-time RP id
    - `scdb` metadata (e.g., disposition, partyWinning, Fowler score) if required by specs

#### 3.1.2 `p2.json` (“authorities payload” / shipped RP) — candidate visible only when admitted
Representative (see `legal-10/datasets/eus/legal10_3step_v1/eus/eu__1826-018/p2.json`):
- Top-level:
  - `type: "authorities"`
  - `payload_id: "p2"`
  - `payload_version`
  - `candidate_visible: true`
  - `content.authorities[]` where each authority includes:
    - `authority_id` (stable id; can align with p1 citation ids)
    - `source` (`SCOTUS` / `CAP` / etc.)
    - `usCite` and/or `capCite`
    - `caseName`
    - `text` (authority excerpt/full text)
    - `char_count`
    - `inventory_normalized_cite` (canonical cite string used for integrity checks)
    - `ranking` object (e.g., `rank`, `fowler_score`, `pagerank_percentile`)

#### 3.1.3 `ground_truth.json` — runner-only; MUST never be exposed to evaluated model
Representative (see `legal-10/datasets/eus/legal10_3step_v1/eus/eu__1826-018/ground_truth.json`):
- Required keys (M1):
  - `anchor_inventory_full: string[]` — sorted unique normalized citations from anchor
  - `rp_subset: string[]` — sorted unique normalized citations shipped in `p2.json`
  - `known_authority` (object):
    - `controlling_authority: string`
    - `in_favor: string[]`
    - `against: string[]`
    - `most_frequent: string`
- Additional metadata:
  - `anchor_caseId`, `anchor_usCite`, `eu_id`, `eu_version`
  - `provenance` paths/ids (build-time lineage; SHOULD use portable relative paths)

### 3.2 Benchmark packet (`benchmark/…`) — what defines the chain

#### 3.2.1 `benchmark/benchmark.json`
This is benchmark metadata plus the system message used by the runner.
Example as-built exists at `legal-10/runspecs/3-STEP-RUN/benchmark/benchmark.json`.

Required keys (M1 + platform prompt protocol):
- `benchmark_id`, `benchmark_name`, `version`, `description`, `created_at`
- `step_count: 3`, `payload_count: 2`
- `system_message: string` (the runner MUST pass as the initial system message)

#### 3.2.2 `benchmark/plan.json`
Defines orchestration and payload admission. Example: `legal-10/runspecs/3-STEP-RUN/benchmark/plan.json`.

Required keys:
- `benchmark_id`, `plan_id`, `plan_version`
- `steps[]` (array order is execution order), each step includes:
  - `step_id` (3-step MVP: `d1`, `d2`, `j3`)
  - `step_file` (path under benchmark dir)
  - `scoring` (`deterministic` or `judge`)
  - `output_contract` (string id; used for validation/reporting)
  - `inject_payloads: string[]` (payload ids that become admitted at that step)
  - deterministic steps: `scorer_ref`
  - judge steps: `judge_prompt_file`
  - `judge_grades_step_ids` appears on the final step to specify which IRACs are graded together

#### 3.2.3 `benchmark/model_steps/{d1,d2,j3}.json`
Each step definition includes:
- `step_id`, `step_name`
- `prompt_template` (with placeholders)
- `placeholders[]` listing allowed placeholder names
- `output_schema` (JSON-schema-like object describing required output keys)
- `output_contract` id

**Contract enforcement requirement:** the runner MUST validate parsed outputs against each step’s `output_schema` before carry-forward.

#### 3.2.4 `benchmark/judge_prompts/irac_mee_pair_v1.json`
Defines the judge protocol for grading both d2 and j3 IRAC outputs in a single call (see `legal-10/docs/_essentials/fdq/post/irac-pair-scoring.md`).

### 3.3 RP build-time package (intermediate; not shipped in released bundle)
Under `legal-10/datasets/rps/`, per-RP directories contain:
- `payloads/d1.json` (anchor + citations roster)
- `payloads/d2.json` (authorities)
- `doc3.json` (optional)
- `manifest.json` (per-RP manifest with hashes)

Example: `legal-10/datasets/rps/rpv1__1826-018/manifest.json`.

**Relationship to shipped EU:** EU `p1.json` and `p2.json` are derived from RP payloads and/or shared dataset sources; the RP directory itself is not shipped in the released benchmark bundle.

### 3.4 Bundle sealing (released bundle integrity)
Per M1, sealing is **non-negotiable** for the released bundle.

#### 3.4.1 `manifest.json` schema (bundle root)
Required shape (M1):
```json
{
  "protocol_version": "bundle-manifest-v1",
  "bundle_id": "legal10_3step_v1",
  "created_at": "2026-01-24T00:00:00Z",
  "generator_version": "benchmark-builder.py@<git_sha_or_semver>",
  "files": [
    { "path": "benchmark/benchmark.json", "sha256": "<hex>", "bytes": 123 }
  ]
}
```

Constraints:
- Manifest MUST exclude `manifest.json` and `signature.json` (no self-reference).
- `path` values MUST be normalized relative paths using `/` separators; MUST reject absolute paths, `..`, and backslashes.
- Hashes are SHA-256 over raw file bytes (no JSON canonicalization).
- Ordering MUST be deterministic.

#### 3.4.2 `signature.json` schema (bundle root)
- Detached Ed25519 signature over the exact bytes of `manifest.json` (UTF-8).
- MUST include at minimum: `algorithm`, `key_id`, `signature` (base64 or hex; standardized).
- Private signing key MUST be provided at build time; private keys must not be committed.

#### 3.4.3 Runner verification
The runner MUST refuse to run if:
- any file’s hash/byte count differs from `manifest.json`, or
- `signature.json` fails verification.

### 3.5 How RP, EU, and benchmark artifacts are assembled (build-time)

This section describes the build-time assembly logic at the level needed to implement and verify the MVP toolchain (not just schemas).

#### 3.5.1 Build-time inputs (local, no network)
Build-time uses local dataset artifacts under `legal-10/datasets/`, including (non-exhaustive):
- A DuckDB database containing SCDB/CAP metadata and derived tables (see `legal-10/docs/build-pipeline/sc-dataset-db-schema.md`).
- A citation inventory describing anchor→cited edges and normalized citations (see `legal-10/docs/build-pipeline/data-pipeline-reference.md` and M1 “Required computations”).

#### 3.5.2 RP builder (produces `datasets/rps/rpv1__<anchor_caseId>/…`)
Authoritative references:
- `legal-10/docs/build-pipeline/rp-builder-reference.py.md`
- `legal-10/scripts/rp_builder.py` (as-built implementation)

Required outcomes per RP:
- Produce payloads (at minimum) that support later EU packaging:
  - a d1-style anchor payload (anchor text + citation roster)
  - a d2-style authorities payload (admitted authorities text + ranking metadata)
  - optional supplemental payloads (e.g., `doc3.json`) if included in your benchmark design
- Produce an RP-local `manifest.json` with hashes for RP reproducibility (example: `legal-10/datasets/rps/rpv1__1826-018/manifest.json`).

Selection/ranking requirements (high level):
- Authorities included in the RP must be derivable deterministically from the underlying dataset, using documented tie-breakers and stable ordering (no sampling unless controlled by `--seed`).
- Authority records must include the normalized-cite key used later for `rp_subset` derivation (`inventory_normalized_cite`).

#### 3.5.3 EU builder (produces `datasets/eus/<benchmark_id>/eus/<eu_id>/{p1,p2,ground_truth}.json`)
Authoritative references:
- M1 EU schema sections in `legal-10/docs/_essentials/mvp/M1-buildtime-packaging-sealing-dev-brief.md`
- `legal-10/docs/build-pipeline/eu-builder-reference.py.md` and `legal-10/docs/build-pipeline/eu-builder-notes.md`
- `legal-10/scripts/eu_builder.py` (as-built implementation)

Required computations (M1):
- `anchor_inventory_full`: sorted unique normalized cites from `datasets/citation_inventory.parquet` where `anchor_caseId = <anchor_caseId>`.
- `rp_subset`: from shipped `p2.json` authorities: take each authority’s `inventory_normalized_cite`, dedupe, sort.
- `known_authority`: computed deterministically per `legal-10/docs/_essentials/fdq/01-ka-sc.md` (tie-breakers included).

EU assembly requirements:
- `p1.json` and `p2.json` are candidate-visible payloads.
- `ground_truth.json` is runner-only and MUST NOT be staged or otherwise made visible to the evaluated model.

#### 3.5.4 Benchmark builder (produces `runspecs/3-STEP-RUN/benchmark/…`)
Authoritative references:
- M1 benchmark packet sections (plan/steps) in `legal-10/docs/_essentials/mvp/M1-buildtime-packaging-sealing-dev-brief.md`
- `legal-10/docs/mvp/3-step-run-benchmark-structure.md`
- `legal-10/runspecs/3-STEP-RUN/benchmark_builder.py` (as-built implementation)

Benchmark builder MUST:
- Materialize `benchmark.json` (including `system_message`), `plan.json`, step files, and judge prompt file.
- Ensure step output schemas match the FDQ-defined response contracts.

#### 3.5.5 Bundle sealer (produces bundle-root `manifest.json` + `signature.json`)
Authoritative reference: sealing section of M1 (`legal-10/docs/_essentials/mvp/M1-buildtime-packaging-sealing-dev-brief.md`).

Sealer MUST:
- Walk the bundle root deterministically, hashing all benchmark + EU files.
- Emit `manifest.json` with normalized relative `/` paths.
- Emit `signature.json` as Ed25519 detached signature over `manifest.json` bytes.

## 4) Runner semantics (3-step MVP)

### 4.1 Inputs and execution contract
Runtime runner takes:
- `benchmark_dir`: path to `benchmark/` directory
- either `eu_dir` (single EU) or `eu_root` (multiple EUs)
- `runs_dir`: output root for run artifacts
- evaluated model adapter config (provider/model)
- judge model adapter config (provider/model)

### 4.2 Session strategy and statefulness
Baseline requirement: **Replay_Minimal**:
- Each step is a fresh model call with fresh messages built from:
  - benchmark `system_message`
  - admitted payloads (p1 always; p2 only when admitted)
  - sanitized `candidate_state` (carry-forward artifacts)
- No chat history is accumulated between steps beyond these artifacts.

### 4.3 Payload admission control (plan-driven)
- Admission is controlled by `plan.json` `inject_payloads`.
- Admission is **cumulative**: once admitted, a payload remains visible for subsequent steps.
- Referencing an unadmitted payload in prompt placeholders is a runtime error.

### 4.4 Staging and isolation (no-leak)
Before each evaluated model call:
1. Create `staging/<run_id>/<call_id>/`.
2. Write/copy ONLY:
   - the step definition for this call (current step)
   - admitted payload files for this call
   - sanitized `candidate_state.json`
3. Ensure `ground_truth.json`, judge prompts, future steps, and unadmitted payloads are not present in staging.
4. Emit audit hashes for all staged files.
5. Build messages per the prompt window protocol.
6. Call evaluated model adapter.
7. Parse + validate output; update sanitized state.
8. Emit run.jsonl + audit_log.jsonl records.
9. Delete staging directory.

### 4.5 Prompt window protocol (message assembly)
Per `legal-10/docs/platform/prompt-messages.md`:
- Each “window” MUST be a separate `{"role":"user","content": ...}` message.
- Window ordering MUST be:
  1) ENV  
  2) ANCHOR_PACK  
  3) EVIDENCE_PACK (only when p2 admitted)  
  4) CARRY_FORWARD (only if non-empty)  
  5) TASK  
  6) OUTPUT_GUARD
- Fencing format MUST be:
  - `<<<BEGIN_{NAME}>>>` … `<<<END_{NAME}>>>`

### 4.6 Output parsing and contract validation
For each step:
- Runner MUST parse model output to JSON.
- Runner MUST validate parsed JSON against the step’s `output_schema`.
- If parse/validation fails:
  - Runner MUST record the failure in run artifacts.
  - Runner MUST avoid propagating invalid fields into `candidate_state`.
  - Deterministic scoring steps MUST assign score `0.0` on invalid output (unless spec says otherwise).

### 4.7 Scoring and post-chain steps
- d1 (KA-SC): deterministic scoring (see scorer spec + implementation).
- d2 + j3: judge-graded as a pair by a separate judge call.
- Citation integrity: deterministic post-chain evaluation that:
  - extracts citations from IRAC outputs
  - checks closed-book citations against `anchor_inventory_full`
  - checks open-book citations against `anchor_inventory_full ∪ rp_subset`
  - computes RP-usage metrics

### 4.8 Run artifacts (required)
Minimum required outputs per run:
- `run.jsonl`
- `audit_log.jsonl`
- `candidate_state.json`
- `summary.json`
- `run_manifest.json`

Recommended additional artifact:
- `trace.jsonl` (structured trace events; see `inter-step-requirements.md` IS-5.4.*)

## 5) Scoring specifications (3-step MVP)

### 5.1 d1 — Known Authority (KA-SC)
Authoritative spec:
- `legal-10/docs/_essentials/fdq/01-ka-sc.md`
- `legal-10/docs/mvp/d1-known-authority-scorer.py.md`

Key requirements:
- Output contains exactly 4 keys: `controlling_authority`, `in_favor`, `against`, `most_frequent`.
- Ground truth uses deterministic SQL/tie-breaks (Fowler score, occurrences, lexicographic).
- Score is the mean of four 0.25-weight components:
  - exact match: controlling_authority
  - F1: in_favor
  - F1: against
  - exact match: most_frequent

### 5.2 Judge IRAC pair grading (d2 + j3)
Authoritative spec: `legal-10/docs/_essentials/fdq/post/irac-pair-scoring.md`.

Key requirements:
- A single judge call grades both IRACs.
- Output schema version: `irac_mee_pair_v1`.
- Rubric: Issue/Rule/Application/Conclusion scored 0–6 each (total 24 per IRAC).
- Runner computes totals and normalized scores from judge grades.

### 5.3 Citation integrity (post-chain)
Authoritative specs:
- `legal-10/docs/_essentials/fdq/post/citation_integrity.py.md`
- `legal-10/docs/mvp/post-citation-integrity.py.md` (reference; may lag implementation)

Key requirements:
- Closed-book validity: citations must be in `anchor_inventory_full`.
- Open-book validity: citations must be in `anchor_inventory_full ∪ rp_subset`.
- Primary extraction should use explicit `citations` field; fallback to text fields if missing.

- **SRL-RUN-IS-3.1.3 (MUST):** Stage ONLY admitted payloads.

#### 6.2.4 Message assembly (IS-4.* / R5.*)
- **SRL-RUN-R5.1 (MUST):** Use fenced windows.
- **SRL-RUN-R5.2 (MUST):** Window order fixed.
- **SRL-RUN-R5.3 (MUST):** Support Replay_Minimal at minimum; record session strategy in run manifest.

#### 6.2.5 Audit + artifacts (IS-5.* / R7.*)
- **SRL-RUN-IS-5.1.2 (MUST):** Record staged file hashes.
- **SRL-RUN-IS-5.1.3 (MUST):** Record message hash (and preferably byte count).
- **SRL-RUN-R7.3 (MUST):** `run_manifest.json` records provenance including runner version, session strategy, and hashes of run artifacts.
- **SRL-RUN-R7.4 (MUST):** `summary.json` provides deterministic rollups (including aggregate score and chain completion status).
- **SRL-RUN-IS-5.4.1 (SHOULD):** Emit `trace.jsonl` structured events.

## 7) Objective assessment: as-built status vs this spec (what is implemented vs needs development)

This section assesses the current as-built implementation under `legal-10/runspecs/3-STEP-RUN/` against the MVP spec sources listed in §0.

### 7.1 What is implemented (MET)
- Core runner executes steps per `benchmark/plan.json`: `legal-10/runspecs/3-STEP-RUN/run_3s.py`.
- Payload admission by `inject_payloads`: `legal-10/runspecs/3-STEP-RUN/runtime/payload_gate.py`.
- Staging directory creation and cleanup: `legal-10/runspecs/3-STEP-RUN/runtime/staging.py`.
- Windowed message assembly (ENV/ANCHOR_PACK/EVIDENCE_PACK/CARRY_FORWARD/TASK/OUTPUT_GUARD): `legal-10/runspecs/3-STEP-RUN/runtime/input_assembler.py`.
- Candidate state sanitization (forbidden exact keys + prefixes): `legal-10/runspecs/3-STEP-RUN/runtime/state.py`.
- Deterministic d1 scoring: `legal-10/runspecs/3-STEP-RUN/scorers/d1_known_authority_scorer.py`.
- Citation integrity with open-book union validity: `legal-10/runspecs/3-STEP-RUN/scorers/citation_integrity.py`.
- Audit log emission with staged file hashes + message hash: `legal-10/runspecs/3-STEP-RUN/runtime/audit.py`.

### 7.2 Implemented but incomplete (PARTIAL)
- **Messages are not assembled from staged bytes** (audit purity gap): messages are built from in-memory payload dicts rather than re-reading the staged files (`run_3s.py` + `input_assembler.py`), while staging writes JSON re-dumps without `sort_keys` (`staging.py`).
- **Payload placeholder semantics are limited**: current `_resolve_placeholders` supports only `{anchor_*}` and `{research_pack_content}`; it does not support generic `{p1.*}` / `{p2.*}` dot/bracket paths required by IS-2.2.*.
- **Determinism and canonicalization details**: staging writes JSON without `sort_keys=True`. Builders appear deterministic, but end-to-end “byte-identical bundle builds” require verifying timestamps and ordering across RP/EU/benchmark builders.

### 7.3 Missing (MISSING / not fully developed)
- **Bundle sealing (manifest + signature) and runtime verification**: required by M1 for released bundles (§3.4). Current runspec runner accepts benchmark/eu dirs directly and does not verify a bundle manifest/signature.
- **Output contract validation**: runner parses JSON but does not validate against step `output_schema`. Evidence: the sample run output `legal-10/runs/run_20260208_080028_154291/candidate_state.json` shows d2 missing required `citations` field even though `benchmark/model_steps/d2.json` requires it.
- **`trace.jsonl` emission**: present in platform requirements but not implemented in 3-step runner.

### 7.4 Known spec-vs-artifact drift (concrete dates)
- The sample run directory `legal-10/runs/run_20260208_080028_154291/` (dated 2026-02-08) contains older `summary.json` and `run_manifest.json` schemas that do **not** include newer fields currently written by `legal-10/runspecs/3-STEP-RUN/run_3s.py` (e.g., `aggregate_score`, `chain_completion`, `session_strategy`, `file_hashes`, `reproducibility_key`).
