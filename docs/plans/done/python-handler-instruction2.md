Background: BlockData is a document processing platform with a FastAPI execution plane (services/platform-api/) that runs Python plugins. We are absorbing Kestra's open-source plugin ecosystem — 131 Java plugin repos at E:/kestra-io/plugins/ with 945 catalog entries already seeded in the database with full schemas (inputs, outputs, examples). A runtime substrate was just built: auth_providers.py (6 auth patterns with auto-detection), serialization.py (msgspec JSONL + chunked bulk writes), and an enhanced ExecutionContext with file I/O, temp files, Jinja2 rendering, and cleanup. Three plugins (GCS, ArangoDB, HTTP) are working on the substrate as reference implementations. The plugin contract is: subclass BasePlugin, declare task_types, implement async run(params, context) -> PluginOutput, call resolve_auth(creds) for authentication, use encode_jsonl/decode_jsonl/chunked_write for data, and context.download_file/context.upload_file for artifacts.

What's needed next: Hand-translate a small batch of plugins onto the substrate — starting with either MongoDB (E:/kestra-io/plugins/plugin-mongodb/, 7 RunnableTask classes) or the next GCP service (BigQuery or Firestore) — while documenting the exact file structure, import patterns, auth usage, parameter mapping, and output shape as you write each one. That documentation becomes the template for a generator that reads from the catalog data and Java source to produce plugin files at scale. The key source files to read are: app/domain/plugins/models.py (contract), app/plugins/gcs.py and app/plugins/arangodb.py (reference implementations), app/infra/auth_providers.py (auth), app/infra/serialization.py (data utilities), and the Kestra Java source for whichever plugin family is being translated.
