# Docling MD Preview Plan Assessment

- Source plan: `docs/parse/docling-md-preview-plan.md`
- Date reviewed: `2026-03-13`
- Reviewer: `Codex`
- Verdict: `Fail`

## What This Is

This is a styling-focused implementation plan for the Parse workbench's Docling MD tab. The intent is good: split the Parse-specific markdown surface away from the shared markdown preview class, then port the prose look-and-feel from `web-docs` so the rendered markdown feels closer to the Starlight docs experience.

## Findings

### Critical

None.

### Major

#### 1. The light-mode inline-code port cannot match the stated source as written

The plan says the Docling MD preview should match the `web-docs` prose system and explicitly includes light-mode verification. But the inline-code background port hardcodes the dark-theme `--sl-color-bg-inline-code` recipe from `web-docs/src/styles/global.css:85` into the proposed `.docling-md-preview code:not(pre code)` rule at `docs/parse/docling-md-preview-plan.md:211-215`. The source system has a different light-theme override at `web-docs/src/styles/global.css:156`, and the final inline-code rule resolves through that variable at `web-docs/src/styles/global.css:342`.

As written, dark mode may be close, but light mode will not be source-aligned. That is a direct contradiction with the stated goal and with the verification step at `docs/parse/docling-md-preview-plan.md:300`.

#### 2. The scope split is underspecified against the current Docling MD markup

The plan says the Parse Docling MD tab should stop using `parse-markdown-preview` and switch to `docling-md-preview`, calling this "the only component change" at `docs/parse/docling-md-preview-plan.md:121`. But the current container in `web/src/pages/useParseWorkbench.tsx:274` has two classes:

`parse-docling-md-preview parse-markdown-preview px-6 py-4`

The plan only discusses renaming `parse-markdown-preview` and never states whether `parse-docling-md-preview` should remain, be removed, or gain any meaning. That makes the post-change selector contract ambiguous and leaves cleanup to implementer judgment.

### Minor

#### 3. The font-loading step is not actually surface-scoped

The plan correctly says changing the app-wide font is out of scope at `docs/parse/docling-md-preview-plan.md:48`, but Step 2 imports Plus Jakarta Sans in `web/src/main.tsx` via `docs/parse/docling-md-preview-plan.md:71-79`. That keeps the rendered font usage scoped to `.docling-md-preview`, but it still changes the asset payload for the whole app, not just the Parse Docling MD tab.

This is not a blocker by itself, but the plan should acknowledge it as an app-wide dependency and performance tradeoff instead of presenting it as purely local.

#### 4. Verification is too manual for a shared-surface split

The verification section at `docs/parse/docling-md-preview-plan.md:297-302` is entirely manual. For a plan whose main safety claim is "Parse-only surface changes, shared preview untouched," that is thin. A focused regression check should verify that:

- the Parse Docling MD tab renders with the new class contract
- `PreviewTabPanel` still renders the shared `parse-markdown-preview` path
- light and dark theme token values resolve as expected for inline code

## Specific Gaps, Contradictions, and Ambiguity

- The source-alignment claim conflicts with the proposed light-mode inline-code rule.
- The plan describes one class rename, but the real markup currently carries two classes.
- The plan treats font adoption as scoped behavior while using a global app entrypoint import.
- The plan does not define a regression test strategy for the scope boundary it depends on.

## Required Changes Before Implementation

1. Rewrite the inline-code background section so it preserves the theme-specific `--sl-color-bg-inline-code` behavior from `web-docs`, including the light-mode override.
2. Update the component-change section to match the real markup in `useParseWorkbench.tsx`. State explicitly whether `parse-docling-md-preview` stays or is removed.
3. Document the operational tradeoff of loading Plus Jakarta Sans from `web/src/main.tsx`, or choose a different loading strategy if global asset cost is a concern.
4. Add at least one focused regression test or DOM assertion that proves the Parse Docling MD tab and the shared markdown preview now use different class contracts.

## Verification Expectations

Before implementation should be considered ready:

- Light and dark mode inline-code backgrounds must resolve to the same formulas used by `web-docs`.
- The Parse Docling MD tab must render only the intended post-change class contract.
- The generic markdown preview path in `PreviewTabPanel` must remain unchanged.
- The app should still build after the font dependency is added and imported.
