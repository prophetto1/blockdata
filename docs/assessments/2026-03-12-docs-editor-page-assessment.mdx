# Docs Editor Page Assessment

- Source plan: `docs/plans/2026-03-12-docs-editor-page.md`
- Date reviewed: `2026-03-12`
- Reviewer: `Codex`
- Verdict: `Fail`

## What This Is

This plan proposes a new standalone `/app/docs` page that reuses the existing OnlyOffice integration outside the ELT workbench. The product intent is reasonable. The current shell and project-focus behavior do not support the plan's stated interaction model yet.

## Scope

The plan covers three implementation areas:

1. A new `DocsEditor` page with a document list and editor panel.
2. Route and navigation registration for `/app/docs`.
3. Manual verification in the browser.

That is the right surface area for a first pass, but the plan leaves out one shell integration change and misstates how project switching would work.

## Findings

### Critical

#### 1. The core project-switching model is incompatible with the current shell behavior

The plan says project selection for `/app/docs` will rely on `useProjectFocus()` plus the shell `ProjectSwitcher`. See [`2026-03-12-docs-editor-page.md:7`](/home/jon/BD2/docs/plans/2026-03-12-docs-editor-page.md#L7) and [`2026-03-12-docs-editor-page.md:237`](/home/jon/BD2/docs/plans/2026-03-12-docs-editor-page.md#L237).

That is not how the current shell behaves. The existing switcher always navigates to `/app/elt/${project.value}` after changing focus. See [`ProjectSwitcher.tsx:65`](/home/jon/BD2/web/src/components/shell/ProjectSwitcher.tsx#L65). In other words:

- a user cannot stay on `/app/docs` and switch projects from the shell
- the plan's "standalone" route would immediately hand control back to ELT when the user changes project
- the manual smoke test does not exercise the primary cross-project workflow the architecture depends on

This is a blocking contract mismatch. The plan must either:

- make `/app/docs` explicitly project-scoped, or
- change the shell switcher behavior so it preserves the current route when a route supports focused-project switching

### Major

#### 2. The plan omits the app-shell layout change needed for a full-height editor workspace

The proposed page uses a full-height editor layout at [`2026-03-12-docs-editor-page.md:91`](/home/jon/BD2/docs/plans/2026-03-12-docs-editor-page.md#L91). But the shell currently only locks main scroll for a small allowlist of workspace-style routes such as ELT, editor, settings, API editor, marketplace detail, and superuser. `/app/docs` is not included. See [`AppLayout.tsx:154`](/home/jon/BD2/web/src/components/layout/AppLayout.tsx#L154) and [`AppLayout.tsx:162`](/home/jon/BD2/web/src/components/layout/AppLayout.tsx#L162).

That omission matters because the shell's main region stays `overflow: auto` when a route is not in that allowlist. See [`AppLayout.tsx:195`](/home/jon/BD2/web/src/components/layout/AppLayout.tsx#L195). For a page that embeds a full-height OnlyOffice editor, that creates a real risk of nested scroll behavior and unstable height calculations.

The plan should explicitly decide whether `/app/docs` is:

- a true workspace route that must be added to the shell's scroll-lock rules, or
- a normal scrolling content page, in which case the proposed full-height layout should be redesigned

#### 3. Verification is too weak for a route whose behavior depends on shell state and persisted project focus

The verification section is limited to TypeScript compile and a manual smoke pass. See [`2026-03-12-docs-editor-page.md:153`](/home/jon/BD2/docs/plans/2026-03-12-docs-editor-page.md#L153) and [`2026-03-12-docs-editor-page.md:211`](/home/jon/BD2/docs/plans/2026-03-12-docs-editor-page.md#L211).

That is not enough here because the route is not self-contained. It depends on:

- `useProjectFocus()` persistence and fallback behavior at [`useProjectFocus.ts:44`](/home/jon/BD2/web/src/hooks/useProjectFocus.ts#L44)
- automatic first-project selection at [`useProjectFocus.ts:124`](/home/jon/BD2/web/src/hooks/useProjectFocus.ts#L124)
- shell-level navigation behavior in the project switcher at [`ProjectSwitcher.tsx:65`](/home/jon/BD2/web/src/components/shell/ProjectSwitcher.tsx#L65)
- full-height shell layout behavior in [`AppLayout.tsx:195`](/home/jon/BD2/web/src/components/layout/AppLayout.tsx#L195)

At minimum, the plan should require focused tests or a broader verification pass for:

- route entry with no prior focused project
- route entry with a persisted focused project
- project switching while on `/app/docs`
- editor sizing and scroll behavior inside the shell

### Minor

#### 4. The no-project UX described in the page conflicts with the hook's current fallback behavior

The page implementation includes a "Select a project to view documents" empty state at [`2026-03-12-docs-editor-page.md:82`](/home/jon/BD2/docs/plans/2026-03-12-docs-editor-page.md#L82). But `useProjectFocus()` currently auto-selects the first project whenever there is no active or persisted focus and projects are available. See [`useProjectFocus.ts:124`](/home/jon/BD2/web/src/hooks/useProjectFocus.ts#L124).

That means the plan is describing two different UX models:

- explicit user selection before document access
- implicit fallback to the first available project

This is not a blocker on its own, but the intended behavior should be stated clearly before implementation starts.

## Required Changes Before Implementation

1. Resolve the project-selection contract. Either make the route project-scoped or update the shell switcher so `/app/docs` can remain active while focus changes.
2. Decide whether `/app/docs` is a workspace-style full-height route and, if so, include the required `AppLayout` shell change in the plan.
3. Rewrite the verification section to cover project-focus persistence, project switching on the docs page, and shell/editor sizing behavior.
4. Clarify the intended no-project experience so the page logic matches the actual `useProjectFocus()` fallback rules.

## Verification Expectations

Before this plan should be considered implementation-ready, the following should be true:

- A reviewer can switch projects from the shell without being forced out of `/app/docs`, or the route contract explicitly requires a project param.
- `/app/docs` renders with the intended full-height editor layout inside the shell without double scrolling or clipped editor space.
- Initial route entry is defined and verified for both persisted-focus and no-focus states.
- The docs list updates for the resolved project and keeps selection behavior predictable when the project changes.
- The route, nav item, and OnlyOffice load path are exercised beyond typecheck alone.

## Final Recommendation

`No-Go` as written.

The page itself is straightforward. The plan is not. Its main flaw is that it treats project context as already solved when the current shell implementation says otherwise. Fix the project-switching contract and shell-layout integration first, then re-review the revised plan before implementation proceeds.
