# OnlyOffice ELT Workbench Integration Plan Assessment

## Plan Metadata

- Source path: `docs/plans/2026-03-11-onlyoffice-elt-workbench-integration.md`
- Date reviewed: `2026-03-12`
- Reviewer: `Codex`

## Verdict

`Fail`

## Findings

### Critical

1. The proposed trust boundary leaks the bridge secret to the browser.
   The plan's `POST /onlyoffice/config` response embeds `?token={ONLYOFFICE_CALLBACK_TOKEN}` in both the document URL and the callback URL. That token is then visible in browser memory and network tooling, which collapses the separation between browser-facing routes and container-facing routes. Any authenticated user who obtains that token can call `/onlyoffice/doc/{session_id}` and `/onlyoffice/callback/{session_id}` directly. The bridge needs per-session, short-lived signed access scoped to one session, not a shared static secret passed through the client.

2. Session authorization stops at `open`, not at ongoing session use.
   The plan checks ownership only when `/onlyoffice/open` resolves `source_uid`, but `/onlyoffice/config` reads a session file by `session_id` and returns a live editor config without re-checking that the caller owns that session. The session metadata shown in the plan does not store `subject_id`, `project_id`, or any other browser-session binding, so a leaked or guessed `session_id` becomes enough to attach to another user's edit session. This is a direct authorization gap in the core edit path.

### Major

1. The overwrite path has no concurrency or conflict strategy.
   The callback flow always writes the edited bytes back to the original `source_locator` via upsert. The plan records a content hash, but it never uses that hash to detect that the source changed in storage after the session opened. Two editors, or an editor plus a replacement upload, can silently clobber each other. The plan needs explicit last-writer policy, optimistic concurrency check, or session locking.

2. The plan says cached files live only for the duration of an editing session, but it defines no cleanup lifecycle.
   There is no close endpoint, TTL, sweeper, or startup cleanup for the cache directory. That leaves stale `.docx` files and session metadata on local disk indefinitely, increases the chance of stale-session reuse, and creates an unbounded storage growth path on the platform-api host.

3. The preview refresh loop is underspecified and likely stale in the current UI architecture.
   `PreviewTabPanel` resolves a signed URL when `doc` changes and reuses that URL for preview rendering. The plan expects "Preview" to show the updated document immediately after edit, but it does not define how the frontend invalidates the old signed URL or forces `DocxPreview` to refetch fresh bytes after a save callback. Without that, the core "edit, save, preview updated content" loop is not reliable.

4. Verification coverage is backend-heavy and misses the highest-risk frontend behavior.
   The proposed tests cover the bridge routes, but they do not add frontend tests for the `.docx` edit toggle, editor mount/unmount lifecycle, preview refresh after save, or failure-state recovery in `PreviewTabPanel`. That leaves the primary user-visible behavior unguarded.

### Minor

1. The plan's manual verification section contradicts its API contract.
   The `open` endpoint contract is defined as `source_uid` only, but the manual test sequence later says the browser posts both `source_uid` and `source_locator`.

2. Task `0.2` still carries an outdated commit message.
   The plan correctly removes the old `python-multipart` dependency requirement, but the example commit message still says `add PyJWT and python-multipart`.

3. The proposed `platformApiFetch` helper repeats the same access-token acquisition logic already present in `web/src/lib/edge.ts`.
   The cleanup direction is right, but this draft still creates another copy of the token-refresh flow instead of extracting a single shared auth helper.

## Specific Gaps, Contradictions, And Ambiguity

- The plan treats `ONLYOFFICE_CALLBACK_TOKEN` as a container-only credential, but the proposed config payload exposes it to the browser.
- The plan says the local cache is session-scoped, but it does not define how a session ends or how stale cache entries are removed.
- The plan stores a `content_hash`, but it does not define any behavior that uses it to prevent overwriting newer storage state.
- The verification narrative assumes preview invalidation after save, but no invalidation mechanism is specified in the frontend tasks.

## Required Changes Before Implementation

1. Replace the shared callback-token query-param design with a per-session, short-lived signed access model that does not expose a reusable bridge secret to the browser.
2. Store opener identity in session metadata and enforce it on every browser-facing route, especially `/onlyoffice/config`.
3. Define the session lifecycle explicitly:
   create, refresh/touch, close, TTL expiry, and cache cleanup.
4. Add a conflict policy for save callback writes before using storage upsert on the original locator.
5. Add an explicit preview invalidation or refetch path so the read-only preview reflects saved edits without a full page refresh.
6. Add frontend verification for toggle behavior, editor teardown, save-to-preview refresh, and failure handling.

## Verification Expectations

- Verify the browser never receives a reusable bridge secret that can be replayed across sessions.
- Verify a second authenticated user cannot use another user's `session_id` to obtain editor config or document bytes.
- Verify concurrent edit or replace scenarios follow an explicit conflict rule instead of silent last-write wins.
- Verify cached session artifacts are removed on close or expiry and do not accumulate across restarts.
- Verify a saved edit is visible in the preview path without relying on manual hard refresh.
- Verify backend smoke tests and at least one frontend integration test cover the inline edit toggle flow.

## Final Recommendation

`No-Go` for implementation as written.

The architecture direction is reasonable: keep editing inside the existing workbench tab and reuse Supabase Storage through a platform-api bridge. But the current plan is not safe enough to execute because it exposes a reusable bridge credential to the browser, does not bind sessions to their opener across the full lifecycle, and does not define conflict handling or cache cleanup. Fix those items first, then re-run a short gate review before more implementation work lands.
