# ELT Assets Panel Upgrade Assessment

- Source plan: `docs/plans/2026-03-12-elt-assets-panel-upgrade.md`
- Date reviewed: `2026-03-12`
- Reviewer: `Codex`
- Verdict: `Fail`

## What This Is

This is an implementation plan to upgrade the ELT workbench assets tree so it behaves more like the superuser workspace file tree. The intended outcome is feature parity for common tree operations: rename, move, delete confirmation, folder delete, drag-to-move, and right-click actions.

## Scope

The plan covers five layers:

1. Database RPCs for `rename_document`, `move_document`, and `move_documents_by_prefix`.
2. A new frontend helper module for document and folder operations.
3. A reusable assets-tree context menu component.
4. A substantial upgrade to `AssetsPanel.tsx`.
5. Hook wiring in `useEltWorkbench.tsx` plus manual verification.

The plan explicitly keeps physical storage paths unchanged and treats folder structure as a logical `doc_title` concern only.

## Findings

### Critical

#### 1. The new `SECURITY DEFINER` RPCs have no ownership or project authorization checks

The proposed SQL updates `public.documents_v2` directly and marks all three functions as `SECURITY DEFINER`, but the function bodies shown in the plan do not validate `auth.uid()`, `owner_id`, or project ownership before mutating rows. See the plan SQL at [`2026-03-12-elt-assets-panel-upgrade.md:52`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L52) and [`2026-03-12-elt-assets-panel-upgrade.md:97`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L97).

That is not a theoretical concern. The current schema only shows a `SELECT` RLS policy on `documents_v2`, not update/delete policies that would constrain a definer function automatically. See [`20260208022131_003_v2_parallel_documents_blocks.sql:82`](/home/jon/BD2/supabase/migrations/20260208022131_003_v2_parallel_documents_blocks.sql#L82). The projects pattern elsewhere in the repo explicitly joins back to owned projects when auth matters; this plan does not. See [`20260227120000_049_flow_sources.sql:37`](/home/jon/BD2/supabase/migrations/20260227120000_049_flow_sources.sql#L37).

As written, this would let any authenticated caller invoke privileged rename or move operations against rows the function can see, regardless of whether the caller owns the document.

### Major

#### 2. The plan mixes stale and current data contracts

The key facts claim the write target is `source_documents`, with `view_documents` as a view over `documents_v2`. See [`2026-03-12-elt-assets-panel-upgrade.md:30`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L30). The actual repo state does not support that cleanly:

- The real writable table in migrations is `documents_v2`. See [`20260208022131_003_v2_parallel_documents_blocks.sql:7`](/home/jon/BD2/supabase/migrations/20260208022131_003_v2_parallel_documents_blocks.sql#L7).
- The frontend still reads from `TABLES.documents = 'view_documents'` and labels `sourceDocuments = 'source_documents'`. See [`tables.ts:2`](/home/jon/BD2/web/src/lib/tables.ts#L2).
- `fetchAllProjectDocuments` uses `TABLES.documents`, so the ELT UI currently depends on `view_documents` for reads. See [`projectDocuments.ts:43`](/home/jon/BD2/web/src/lib/projectDocuments.ts#L43).

I did not find a migration in this repo creating `view_documents` or `source_documents` as described by the plan. That means the plan is not just using old names casually; it is describing the persistence contract ambiguously enough that someone could implement against the wrong abstraction.

#### 3. The virtual-folder ownership model is internally contradictory

Task 4 first sketches hook-level handlers that call `setVirtualFolders?.(...)` from `useEltWorkbench`. See [`2026-03-12-elt-assets-panel-upgrade.md:585`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L585). A few lines later, the plan notes that virtual folder state currently lives inside `AssetsPanel` and recommends keeping it there instead of lifting it. See [`2026-03-12-elt-assets-panel-upgrade.md:605`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L605).

The current component confirms that `virtualFolders` is local `AssetsPanel` state persisted directly to localStorage. See [`AssetsPanel.tsx:67`](/home/jon/BD2/web/src/components/documents/AssetsPanel.tsx#L67) and [`virtualFolders.ts:5`](/home/jon/BD2/web/src/lib/virtualFolders.ts#L5).

That leaves the implementation path unclear:

- If the hook owns folder rename/delete side effects, the plan must move virtual-folder state out of `AssetsPanel`.
- If `AssetsPanel` owns virtual-folder updates, the hook API and sample code in Task 4 are wrong.

This is a sequencing bug in the plan, not a cosmetic wording issue.

#### 4. Rename, move, and folder-delete semantics are underspecified around conflicts and partial failure

The helper layer describes rename and move as simple `doc_title` rewrites. See [`2026-03-12-elt-assets-panel-upgrade.md:160`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L160) and [`2026-03-12-elt-assets-panel-upgrade.md:180`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L180). The folder delete flow loops through documents, calls `delete_document` one row at a time, and then performs a best-effort storage removal. See [`2026-03-12-elt-assets-panel-upgrade.md:221`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L221).

Missing from the plan:

- What happens if a target logical path already exists.
- Whether moving a folder into itself or a descendant is rejected.
- Whether folder rename/move should be atomic at the database level.
- What the UI should do if some `delete_document` calls succeed and others fail.
- What happens if storage cleanup fails after the database deletes succeed.

Those are user-visible correctness questions. The current plan leaves them to ad hoc implementation decisions.

#### 5. Verification is too weak for a database-plus-interaction change set

The verification section is limited to TypeScript typecheck and a manual checklist. See [`2026-03-12-elt-assets-panel-upgrade.md:648`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L648). That is not enough for a plan that introduces privileged SQL functions and several path-rewrite operations.

Missing verification expectations include:

- SQL-level authorization tests for owned vs unowned documents.
- RPC tests for path rewrites, no-op cases, and invalid inputs.
- Frontend tests for drag/drop behavior and context-menu dismissal.
- Folder-state persistence tests for virtual folders after rename/delete.
- Failure-path tests for partial delete and storage cleanup errors.

### Minor

#### 6. The plan overstates parity with the superuser tree and misstates the rename pattern

The goal claims interaction parity with the superuser workspace tree. See [`2026-03-12-elt-assets-panel-upgrade.md:5`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L5). But the same plan defers Ark UI inline rename and says `window.prompt` matches the superuser pattern. See [`2026-03-12-elt-assets-panel-upgrade.md:405`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L405) and [`2026-03-12-elt-assets-panel-upgrade.md:690`](/home/jon/BD2/docs/plans/2026-03-12-elt-assets-panel-upgrade.md#L690).

The reference component already uses Ark TreeView rename APIs:

- `canRename(...)` at [`WorkspaceFileTree.tsx:384`](/home/jon/BD2/web/src/pages/superuser/WorkspaceFileTree.tsx#L384)
- `onRenameComplete(...)` at [`WorkspaceFileTree.tsx:385`](/home/jon/BD2/web/src/pages/superuser/WorkspaceFileTree.tsx#L385)

This is not a blocker by itself, but the plan should describe the choice accurately: it is aiming for partial interaction parity, not full parity.

## Required Changes Before Implementation

1. Rewrite the RPC section so each function enforces caller ownership explicitly, ideally by joining through `documents_v2.owner_id` or `projects.owner_id` and rejecting unauthorized access.
2. Normalize the data contract section to the repo reality. State exactly which table or view each read and write path uses today, and remove stale `source_documents` language unless this plan also introduces the compatibility layer.
3. Resolve virtual-folder ownership before implementation. Pick one model: either lift virtual-folder state into `useEltWorkbench`, or keep all virtual-folder mutations inside `AssetsPanel` and adjust the hook API accordingly.
4. Add explicit behavior for rename and move conflicts, illegal folder targets, and partial delete or storage cleanup failures.
5. Expand verification to include automated tests for SQL authorization and path-rewrite behavior, plus at least focused frontend tests for context menu, drag/drop, and virtual-folder updates.

## Verification Expectations

Before this plan should be called implementation-ready, the following should be true:

- A migration test or SQL verification proves unauthorized users cannot rename or move another user's documents.
- Rename and move operations reject invalid targets and preserve expected logical paths.
- Folder rename updates both document paths and local virtual-folder state consistently.
- Folder delete behavior is defined and verified for partial failure cases.
- The ELT assets tree behavior is exercised beyond typecheck, either with component tests or an end-to-end UI pass that covers drag/drop and context menu dismissal.

## Final Recommendation

`No-Go` as written.

This plan has the right product intent, but it is not execution-ready. The main problem is not the UI work. It is that the plan couples privileged SQL mutations, stale persistence terminology, and unresolved virtual-folder ownership into one batch without locking down the contract. Tighten those pieces first, then re-review the revised plan before implementation proceeds.
