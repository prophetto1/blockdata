## Verdict

`Fail`

## Findings

### Critical

1. The callback SSRF guard is still bypassable from the browser-scoped session token path.
   The implementation returns a per-session token to the browser in the `document.url` and `callbackUrl` generated by `/onlyoffice/config`, then trusts that token on `/onlyoffice/callback/{session_id}`. That can be safe only if callback URL validation is strict. It is not. The callback handler validates scheme and hostname, but not port, so on the default local topology a caller can present a browser-visible session token and submit `http://localhost:<other-port>/...` as the callback download URL. Because `localhost` still matches the configured host, platform-api will fetch that URL and write the response bytes back to storage. This leaves a real SSRF/write-gadget path open in the most likely dev setup.

### Major

1. The concurrency protection fails open when the verification read fails.
   The new save path tries to compare the current storage hash to the session hash before overwriting, but any exception in that read only logs a warning and then continues with the overwrite. That means transient storage failures, permission issues, or deleted-object races drop the code back to blind last-write-wins behavior. If the implementation is going to advertise optimistic concurrency, the write must stop when the verification step cannot establish current state.

2. The OnlyOffice JWT secret still has a known default, so misconfiguration does not fail closed.
   `Settings.from_env()` populates `onlyoffice_jwt_secret` with `"my-jwt-secret-change-me"`. The bridge then treats any non-empty value as configured. In practice, a deployment that forgets to set `ONLYOFFICE_JWT_SECRET` will still issue valid config JWTs and session tokens with a predictable secret instead of refusing to start or rejecting requests. That is too weak for an auth-bearing secret.

3. Batch 1 has no automated verification for the new bridge.
   The updated plan now includes backend smoke tests, but the implementation reviewed here has not landed them yet. I verified that the FastAPI app imports and registers the four OnlyOffice routes, but there is still no committed test coverage for session ownership, token scoping, callback SSRF rejection, or storage writeback. That is a meaningful gap for a security-sensitive bridge.

### Minor

1. `ConfigRequest.mode` accepts any string.
   The route comment and plan imply `"edit"` or `"view"`, but the implementation does not constrain that value. This is unlikely to be a security issue, but it weakens contract clarity and makes frontend failures less predictable.

## Specific Gaps, Contradictions, And Ambiguity

- The implementation fixes the old shared-secret leak by switching to per-session tokens, but those tokens are still browser-visible and therefore must be treated as browser-reachable credentials, not container-only credentials.
- The callback handler claims SSRF protection, but the current check proves host equality only. It does not prove origin equality.
- The implementation introduces optimistic concurrency language, but its error path still permits overwrite on verification failure.
- The updated plan expects smoke tests for batch verification, but this batch stops before those tests exist.

## Required Changes Before Implementation Continues

1. Tighten callback download URL validation to enforce the full expected origin, not just hostname. At minimum, validate scheme, hostname, and port against `onlyoffice_docserver_internal_url`.
2. Treat the callback route as exposed to a browser-held session credential and harden it accordingly.
3. Change the concurrency guard to fail closed when the current storage state cannot be verified.
4. Remove the hard-coded JWT secret default and require explicit configuration for non-test environments.
5. Land the planned backend smoke tests before batch 2 proceeds.

## Verification Expectations

- Add a regression test that rejects callback download URLs on the correct hostname but the wrong port.
- Add a regression test that rejects save callback processing when the storage verification read fails.
- Add a regression test that proves session tokens are scoped to one session and cannot be replayed across sessions.
- Add a startup or request-path verification that refuses operation when `ONLYOFFICE_JWT_SECRET` is unset or left at the known placeholder value.
- Keep a basic import/openapi check in place to confirm route registration after edits.

## Final Recommendation

`No-Go` for moving on to the next implementation batch until the callback validation bug is fixed.

The batch does incorporate the earlier assessment in meaningful ways: ownership is enforced on `open`, session ownership is re-checked on `config`, and the old static callback secret is gone. Those are real improvements. But the remaining callback trust bug is still blocking, because it turns the browser-visible session token into a practical SSRF/write primitive under the default local topology. Fix that first, add the missing backend smoke tests, and then re-run this gate.
