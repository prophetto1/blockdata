## What This Is

This revision is a substantial improvement over the earlier draft. It adopts the right high-level architecture for this repo:

- Docling remains the parse and structure layer
- extraction runs over stored parse artifacts
- provider, model, and key handling stay in the platform

That shift removes the biggest problem in the first version. The plan is now directionally correct and close to execution-ready, but a few important implementation assumptions are still inaccurate or underdefined.

## Findings

### Critical

None.

### Major

#### 1. The plan overstates what the existing LLM call layer already provides

The new architecture says extraction will run in edge functions using the "existing LLM call infrastructure" and includes a provider matrix for Anthropic, OpenAI, Google, Custom, and Vertex AI (`docs/plans/2026-03-14-extract-feature-design.md:14-29`, `:231-245`).

The current worker code does not provide that abstraction today. It supports two runtime transports:

- `vertex_ai`
- `litellm_openai`

See `supabase/functions/worker/index.ts:34-39` and `:1042-1058`.

It also does not decrypt and forward per-user provider keys for general model execution. Instead, it reads only Anthropic default model settings from `user_api_keys`, while the actual runtime auth comes from platform-managed Vertex or a platform LiteLLM key (`supabase/functions/worker/index.ts:984-1058`).

That means the plan's core direction is good, but the "reuse existing infrastructure" claim is too strong. The implementation still needs an explicit extraction-side provider adapter layer, or the MVP needs to be narrowed to the transports the platform actually has today.

#### 2. Vertex AI fallback is routed through the wrong settings surface

The plan says the Config tab can read from `user_api_keys` plus `providerRegistry.tsx`, and the provider matrix lists "Vertex AI" as a default fallback (`docs/plans/2026-03-14-extract-feature-design.md:21-29`, `:370-384`).

In the current product, Vertex AI is not represented as a normal `user_api_keys` entry. It is managed through `user_provider_connections` and the Google Vertex service-account flow, not the standard API-key form. See:

- `supabase/functions/provider-connections/index.ts`
- `web/src/components/agents/forms/GoogleAuthPanel.tsx`
- `supabase/functions/agent-config/index.ts:62-79`

So the plan's settings and provider-resolution sections do not yet match the real auth model. This needs one explicit decision:

- either MVP extraction uses only API-key-backed providers in `user_api_keys`, or
- the plan adds first-class support for Vertex-backed `user_provider_connections` in the Extract config flow.

#### 3. Page-level extraction is still underplanned operationally

The plan says page-level extraction will load Docling JSON and "split content by page using provenance metadata (same logic as `doclingNativeItems.ts`)," then make one LLM call per page inside a single synchronous edge function (`docs/plans/2026-03-14-extract-feature-design.md:225-247`).

That helper does not currently build page-level reading-order text. It extracts native items and page numbers, but it does not assemble full page payloads suitable for LLM extraction. See `web/src/lib/doclingNativeItems.ts:56-170`.

There is also still an execution-risk mismatch: one call per page inside a synchronous edge function is plausible for short documents, but it can exceed time limits on larger files. The plan notes timeout risk, but leaves chunking as a future enhancement (`docs/plans/2026-03-14-extract-feature-design.md:243-245`).

Before implementation starts, the plan should define one of these:

- a hard page-count or token-size limit for synchronous MVP
- batching and continuation behavior
- or a background-job path for page-level extraction

### Minor

#### 4. The secret helper name is wrong in the backend flow

The plan says the edge function will call `decryptSecret()` from `_shared/api_key_crypto.ts` (`docs/plans/2026-03-14-extract-feature-design.md:220-223`, `:256-266`).

That helper does not exist under that name. The current exported helper is `decryptApiKey()` in `supabase/functions/_shared/api_key_crypto.ts:67-68`.

#### 5. The idempotency note is internally inconsistent

The idempotency section says:

- return existing results if the same `source_uid + schema_uid` already completed
- re-extraction creates a new job

See `docs/plans/2026-03-14-extract-feature-design.md:247`.

Those are both reasonable behaviors, but they need a trigger distinction. For example:

- automatic dedupe for accidental duplicate submits within a short window
- explicit "Run again" action creates a new job

Right now the plan says both without defining when each applies.

## Specific Gaps, Contradictions, and Ambiguity

- The plan now has the right architecture, but it still overclaims current multi-provider reuse.
- Vertex AI fallback is described as if it lived in `user_api_keys`, while the current product routes it through provider connections.
- Page-level extraction depends on page assembly logic that does not exist yet in the cited helper.
- The sync edge-function model still needs clear guardrails for larger documents.
- The idempotency behavior needs one explicit rule.

## Required Changes Before Implementation

1. Rewrite the provider section to distinguish between:
   - what exists today
   - what must be built for extraction

2. Decide the MVP provider surface.
   Either:
   - API-key providers only, or
   - API-key providers plus Vertex service-account connections

3. Add an extraction adapter module to the plan.
   Define where provider-specific calls live and whether they reuse worker helpers, new shared helpers, or a simplified extraction-specific adapter.

4. Define page-level execution limits.
   Add page-count or token limits, or add a background path for larger jobs.

5. Replace `decryptSecret()` with the real helper name and clarify the exact secret lookup path.

6. Clarify dedupe versus explicit re-run behavior.

## Verification Expectations

Before implementation should be considered ready, the revised plan should verify:

- provider resolution for each supported MVP provider
- no secret material in logs or job rows
- successful document-level extraction from stored `markdown_bytes`
- successful page-level extraction from assembled page text, not just raw page tags
- clear rejection or fallback behavior for oversized extraction requests
- parsed-only file visibility in the Extract UI
