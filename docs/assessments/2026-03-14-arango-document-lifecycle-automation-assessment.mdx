## What Changed Between Versions

The newer version is the right direction. It removes the app-facing Arango read API, keeps Arango in a projection role, tightens the reset contract, updates the migration reference to the current RPC definition, and stops treating Arango reconciliation failures as silent success.

The earlier `copy` version is broader and less safe. It expands scope into a new query endpoint, leaves an auth gap in that endpoint, and tells the frontend deletion or reset succeeded even when Arango cleanup failed.

## Findings

### Critical

#### 1. The `copy` version adds an app-facing Arango query endpoint without an ownership check

The proposed `arango-query` function in the `copy` plan authenticates the caller, but it never verifies that the requested `source_uid` belongs to that user before reading from Arango (`docs/plans/2026-03-14-arango-document-lifecycle-automation copy.md:859-910`).

That is a direct data exposure risk. Any authenticated user who can guess or obtain another document's `source_uid` could query its projection. This is also misaligned with the current direction that Arango should remain a projection layer, not an app control surface (`docs/reference/priorities-rewritten.md:10-13`, `docs/reference/priorities-rewritten.md:172-189`).

#### 2. The `copy` version hides reconciliation failure after the source-of-record mutation already succeeded

In the `copy` version, `manage-document` calls the Supabase RPC first, then logs Arango cleanup failures as "non-fatal" and still returns `{ ok: true }` to the client for both delete and reset (`docs/plans/2026-03-14-arango-document-lifecycle-automation copy.md:585-607`, `docs/plans/2026-03-14-arango-document-lifecycle-automation copy.md:610-632`).

Because the RPCs are `SECURITY DEFINER` functions that already mutate Supabase state (`supabase/migrations/20260313210000_080_fix_reset_delete_rpcs_table_names.sql:6-84`), this would knowingly create divergence between Supabase and Arango while reporting success to the UI. That is not an acceptable failure mode for lifecycle management.

### Major

#### 3. The revised version still masks too many Arango failures by treating any `404` as a no-op

The improved plan introduces `deleteBlocksForSourceIfPresent`, but it returns early on any `404` before checking the Arango error body (`docs/plans/2026-03-14-arango-document-lifecycle-automation.md:318-339`). The same broad tolerance appears in the planned document delete and reset helpers (`docs/plans/2026-03-14-arango-document-lifecycle-automation.md:342-359`, `docs/plans/2026-03-14-arango-document-lifecycle-automation.md:368-401`).

That is too broad. A missing blocks collection is a reasonable no-op, but a wrong database, bad path, or broken Arango config can also surface as `404`. The plan should only swallow the specific Arango error codes that represent "document not found" or "collection not found," and it should inspect the response body before deciding the error is safe to ignore.

#### 4. The revised version changes the user-facing mutation path, but it does not require automated tests for the new edge function

The newer plan adds `manage-document` as the new delete/reset entrypoint (`docs/plans/2026-03-14-arango-document-lifecycle-automation.md:593-708`), yet its verification section only requires helper tests and live smoke checks (`docs/plans/2026-03-14-arango-document-lifecycle-automation.md:832-848`).

That leaves the new behavior with no explicit automated coverage for:

- RPC failure mapping
- Arango failure mapping to `502`
- auth handling
- success responses for delete vs reset

The edge function is the main behavioral change in the plan. It should not rely only on manual smoke testing.

#### 5. The `copy` version expands scope beyond the immediate write-side problem

The `copy` version says the goal is to add a minimal read path so the projection is "verifiable and reliable" (`docs/plans/2026-03-14-arango-document-lifecycle-automation copy.md:5`) and adds a full extra task for `arango-query` (`docs/plans/2026-03-14-arango-document-lifecycle-automation copy.md:720-918`).

That scope increase is not needed to solve the stated lifecycle defect. The newer version correctly narrows the plan to the write-side sync contract and operator verification.

### Minor

#### 6. The `copy` version's reset contract does not fully restore the upload-stage shape it claims to want

The `copy` version says reset should return the document to `status: "uploaded"` (`docs/plans/2026-03-14-arango-document-lifecycle-automation copy.md:28`), but its planned patch body clears only conversion fields and leaves parse-derived fields such as `source_total_characters` and normalized `pipeline_config` untouched (`docs/plans/2026-03-14-arango-document-lifecycle-automation copy.md:258-277`, `docs/plans/2026-03-14-arango-document-lifecycle-automation copy.md:316-347`).

The newer version fixes this. That is another reason to use the revised file as the base.

#### 7. The `copy` version points to an outdated RPC migration

Its edge-function context cites migration `20260312190000_078_document_delete_reset_rpcs.sql` (`docs/plans/2026-03-14-arango-document-lifecycle-automation copy.md:542`), but the current corrective definition is in migration `20260313210000_080_fix_reset_delete_rpcs_table_names.sql` (`supabase/migrations/20260313210000_080_fix_reset_delete_rpcs_table_names.sql:1-4`).

This is small, but it is the kind of detail drift that causes implementation confusion during execution.

## Specific Gaps, Contradictions, and Ambiguity

- The `copy` version treats the lack of a read path as part of this plan's scope, while the revised version correctly reframes that as a future concern.
- The `copy` version introduces an app-facing Arango endpoint even though the current architecture direction keeps Arango as a projection layer, not a control or query plane.
- The revised version improves failure handling, but its `404` tolerance is still too permissive and can hide infrastructure errors.
- Neither version defines automated tests for the new `manage-document` edge function, even though that function becomes the new mutation gateway for delete and reset.

## Required Changes Before Implementation

1. Use `docs/plans/2026-03-14-arango-document-lifecycle-automation.md` as the implementation base. Do not implement the `copy` version.
2. Tighten the revised helper design so it ignores only specific Arango "not found" codes, not every `404`.
3. Add explicit automated tests for `supabase/functions/manage-document/index.ts`, including success, RPC failure, auth failure, and Arango reconciliation failure.
4. Keep the read path out of this plan. If it comes back later, require ownership enforcement through Supabase-backed authorization before any Arango lookup.
5. Add a brief reconciliation note for operators that explains what to do when delete or reset returns `502` after Supabase has already succeeded.

## Verification Expectations

Before implementation should be considered ready, the final plan should verify:

- helper tests for upload, parse success, parse failure, zero-block parse, delete, reset, and collection caching
- automated tests for `manage-document` success and failure paths
- that delete and reset surface Arango reconciliation failures to the caller instead of silently succeeding
- that the tolerated no-op cases are limited to specific Arango error codes
- live smoke for upload -> parse -> reset -> delete after the edge function is deployed
